SSO & User Management

// Authentik · Pi Sentry · Google Login · All Services · Nuke System //

Authentik is the single login system for every service on this server. Users log in with their Google (Gmail) account — no new passwords to remember. Authentik runs on the Pi Sentry and is always available, even when the PC is off.

🐉
Architecture: Authentik runs in Docker on the Pi Sentry — always on, always reachable. Caddy (installed via apt on the Pi) uses Authentik's forward_auth endpoint to protect every service. Tailscale routes all external traffic to the Pi's Caddy. The PC being off does not affect SSO at all.

🍓 Runs on Pi Sentry
⚙️ Authentik Installation — Pi Sentry
💡
The full Docker Compose file including Authentik is in raspi.html → Services tab. This section covers the Authentik-specific configuration steps you do after the containers are running.
Summary of what runs where: Authentik server + worker + PostgreSQL + Redis — all Docker containers on the Pi. Caddy — native apt install on Pi (/etc/caddy/Caddyfile).
Step 1 — Deploy Authentik on the Pi (see raspi.html)

Authentik is part of the Pi's main docker-compose.yml. Once the Pi's Docker stack is up, Authentik is already running. Verify:

# SSH into the Pi:
ssh ixora@sentry.ixo.ts.net

docker compose ps | grep authentik
# Should show: authentik-server   running
#              authentik-worker   running
#              authentik-db       running
#              authentik-redis    running

# First-run setup UI (only appears once):
# https://auth.ixo.lol/if/flow/initial-setup/
Step 2 — Create your admin account

Visit on first boot. Set a strong admin password — this is the master key to everything. Store it in Vaultwarden immediately.

Step 3 — Add Google as a Social Login Source

In Google Cloud Console (console.cloud.google.com):

  • New project → "Home Server SSO"
  • APIs & Services → OAuth Consent Screen → External → add your Gmail + all user emails as Test Users
  • Create Credentials → OAuth Client ID → Web application
  • Authorized redirect URI: /source/oauth/callback/google/
  • Save Client ID and Client Secret

In Authentik admin → Directory → Sources → Create → OAuth2/OpenID Source:

Name:             Google
Slug:             google
Consumer Key:     [your Google Client ID]
Consumer Secret:  [your Google Client Secret]
OIDC Well-Known:  https://accounts.google.com/.well-known/openid-configuration
User matching:    Match by email (preferred)
☑ Allow users to change username
Step 4 — Create the Authentik Outpost (for Caddy forward_auth)

Authentik admin → Applications → Outposts → Create:

Name:      caddy-outpost
Type:      Proxy
# Add ALL forward_auth applications to this outpost.
# The outpost exposes the /outpost.goauthentik.io/auth/caddy endpoint
# that Caddy calls on every request.
Step 5 — Configure Caddy to use Authentik

Caddy is installed natively on the Pi via sudo apt install caddy. Edit /etc/caddy/Caddyfile:

# /etc/caddy/Caddyfile

{
    email admin@ixo.lol
}

# Authentik itself — no forward_auth wrapper (it protects itself)
auth.ixo.lol {
    reverse_proxy localhost:9000
}

# Standard protected service — Pi-hosted (same machine → localhost)
jellyfin.ixo.lol {
    forward_auth localhost:9000 {
        uri /outpost.goauthentik.io/auth/caddy
        copy_headers X-authentik-username X-authentik-groups X-authentik-email
        trusted_proxies private_ranges
    }
    reverse_proxy localhost:8096
}

# PC-hosted service — Caddy on Pi reaches PC over Tailscale MagicDNS
movies.ixo.lol {
    forward_auth localhost:9000 {
        uri /outpost.goauthentik.io/auth/caddy
        copy_headers X-authentik-username X-authentik-groups X-authentik-email
        trusted_proxies private_ranges
    }
    reverse_proxy server.ixo.ts.net:8096
}

# TV endpoint — NO forward_auth (Smart TVs can't complete browser SSO flows)
tv.ixo.lol {
    reverse_proxy server.ixo.ts.net:8096
}
# After editing, reload Caddy (no restart needed):
sudo systemctl reload caddy

# Check for config errors first:
caddy validate --config /etc/caddy/Caddyfile

🔐 Caddy forward_auth — Full Template
Pattern A — Pi-hosted service
# Service runs on the Pi → reverse_proxy to localhost
service.ixo.lol {
    forward_auth localhost:9000 {
        uri /outpost.goauthentik.io/auth/caddy
        copy_headers X-authentik-username X-authentik-groups X-authentik-email
        trusted_proxies private_ranges
    }
    reverse_proxy localhost:[PORT]
}

# Admin-only gate (add inside the block):
@notadmin not header X-authentik-groups *authentik-admins*
respond @notadmin "Forbidden" 403
Pattern B — PC-hosted service (GPU/Windows-only)
# Service runs on the PC → reverse_proxy via Tailscale MagicDNS
service.ixo.lol {
    forward_auth localhost:9000 {
        uri /outpost.goauthentik.io/auth/caddy
        copy_headers X-authentik-username X-authentik-groups X-authentik-email
        trusted_proxies private_ranges
    }
    reverse_proxy server.ixo.ts.net:[PORT]   # server.ixo.ts.net = PC's Tailscale hostname
}
⚠️
Pattern B only works when the PC is on. If the PC is off, users will get a 502 Bad Gateway for those services. The forward_auth check itself still works (Pi is up) — it's the upstream proxy target that's unavailable. Pi-hosted services (Pattern A) are unaffected.

🐉
The principle: Every piece of per-user data — emulator saves, reading progress, listening history, MTG decks, AI recommendation history — lives inside D:\Users\[username]\ on the PC, or /mnt/archive/Users/[username]/ on the Pi for Pi-hosted services. The Authentik X-authentik-username header is the key that routes data to the right folder.
🗂️ User Data Architecture

Used by all Pi-hosted services. Lives on the archive drive — counts toward Nextcloud storage quota. Persists independently of whether the PC is on.

🔧 Creating a New User's Folders
On the Pi (SSH in)
# Replace "alice" with the user's Authentik username
USERNAME=alice

# Pi archive folders
mkdir -p /mnt/archive/Users/$USERNAME/{audiobooks,calibre,emulation,nextcloud,inbox}
chown -R 1000:1000 /mnt/archive/Users/$USERNAME

🔐 Adding a New User
  1. Authentik admin → Directory → Users → Create → set username (must match their Google email prefix) + email (exact Gmail address)
  2. Add user to the appropriate group (authentik-family or authentik-admins)
  3. Ask them to visit and sign in with Google — account links automatically on first login
  4. Run the folder creation script above (User Data Storage tab)
  5. For Smart TV users: set a Jellyfin native password (not the Google account — TV apps can't do browser SSO)

💾 Save File Migration Scripts
🐉
The problem: Before Authentik, services created user accounts with local usernames. After Authentik, the username passed to services is the Google account email prefix (e.g. alice from alice@gmail.com). These scripts rename existing data directories to match.
🎮 Emulation Save Files — PC (Dolphin/Xenia)
$oldName = "oldusername"
$newName = "alice"   # Authentik username (Google email prefix)

# Rename the emulation folder
Rename-Item "D:\Users\$oldName" "D:\Users\$newName"

# Fix any symlinks that pointed to the old folder
# (re-run Sunshine pre-launch script for each user)
🍓 Pi Emulation Saves (EmulatorJS)
# SSH into Pi:
OLD=oldusername
NEW=alice

mv /mnt/archive/Users/$OLD /mnt/archive/Users/$NEW
chown -R 1000:1000 /mnt/archive/Users/$NEW
General Migration Principle
💡
For every service: Before enabling Authentik OIDC, update existing user accounts to use the user's exact Gmail address. Most services match on email during OIDC login — existing data is preserved automatically. For save-file-based systems (emulators), use the folder rename scripts above.
💣 Block & Nuke System
🐉
The power of Authentik: Because every service authenticates through Authentik on the Pi, blocking a user in Authentik instantly removes their access from ALL services simultaneously — no need to log into each app individually. The Pi being always-on means this takes effect immediately at any time.
🚫 Temporary Block

Authentik admin → Directory → Users → click user → Deactivate toggle. Session tokens revoked immediately. Re-enable the same way.

💣 Full Nuke — step by step
  1. Authentik admin → Directory → Users → Delete user → immediate access removal everywhere
  2. SSH into Pi: rm -rf /mnt/archive/Users/[username]
  3. On PC: Remove-Item -Recurse "D:\Users\[username]"
  4. Jellyfin admin → Users → delete their account (removes watch history)
  5. Nextcloud: user deletion via docker exec nextcloud php occ user:delete [username]
  6. Block their Tailscale device in the Tailscale admin panel
⚠️
Deletion is permanent. Back up their data first if there's any chance of reinstatement.
🏆 Achievement System — Architecture
🐉
Distributed architecture: Each node runs a lightweight tracking agent that listens for local events (emulator milestones, service usage, media completions). All agents report to the central Achievement Manager on the Pi Sentry (port 9010), which is the authoritative store. The Recommendation Engine on the PC hub reads from the Manager to personalize suggestions.

Adding a new Pi to the cluster? Deploy a tracking agent on it — it automatically registers with the Manager.
🍓 Achievement Manager (Pi — port 9010)
  • Receives signals from all tracking agents
  • Evaluates definitions.json unlock conditions
  • Stores per-user unlock records
  • Feeds the community feed on the dashboard
  • Always available — Pi is always on
📊 Tracking Agent (every node — port 9001/9011/…)
  • Runs on Pi (port 9001), PC hub (port 9011)
  • Future Pis get their own agent + port
  • Monitors local service events
  • Forwards signals to Manager at Pi:9010
  • Lightweight — <40MB RAM per node
Signal flow example
# User alice completes a Pokémon game on EmulatorJS (Pi):
#   EmulatorJS save server → Pi tracking agent (port 9001)
#   Pi tracking agent → Achievement Manager (port 9010) → evaluates definitions
#   Manager → unlocks "Pokémon Master" → updates community feed

# User bob streams a game via Sunshine (PC):
#   Sunshine webhook → PC tracking agent (port 9011)
#   PC tracking agent → Achievement Manager (Pi:9010)
#   Manager → unlocks "Game Streamer" if conditions met
