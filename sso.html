SSO & User Management

// Authentik · Google Login · All Services · Nuke System //

Authentik is the single login system for every service on this server. Users log in with their Google (Gmail) account — no new passwords to remember. You manage who has access to what, all in one place.

⚙️ Authentik Installation (WSL2 + Docker)
🐉
Architecture: Authentik runs in Docker inside WSL2. It exposes port 9000 to Windows. Caddy (running natively on Windows) uses Authentik's forward_auth endpoint to protect every service. Cloudflare Tunnel routes external traffic to Caddy.
Step 1: Install WSL2 + Docker (if not done) 
wsl --install
# Reboot, then open Ubuntu terminal:
sudo apt update && sudo apt install -y docker.io docker-compose
sudo usermod -aG docker $USER
newgrp docker
Step 2: Deploy Authentik 
mkdir ~/authentik && cd ~/authentik

# Download official compose file:
wget https://goauthentik.io/docker-compose.yml

# Create .env file:
cat > .env << 'EOF'
POSTGRES_PASSWORD=change-this-strong-password
AUTHENTIK_SECRET_KEY=change-this-to-50-random-chars
AUTHENTIK_ERROR_REPORTING__ENABLED=false
EOF

docker-compose pull
docker-compose up -d

# Access at http://localhost:9000/if/flow/initial-setup/

Create your admin account on first access. This takes ~2 minutes to fully start.

Step 3: Auto-start Authentik on Windows boot 
# Task Scheduler → Create Task:
# Name: Authentik
# Action: wsl -d Ubuntu -- bash -c "cd ~/authentik && docker-compose up -d"
# Trigger: At startup, delay 2 minutes
# Run whether logged on or not: ☑
Step 4: Add Google as a Social Login Source

In Google Cloud Console (console.cloud.google.com):

  • New project → "Home Server SSO"
  • APIs & Services → OAuth Consent Screen → External → add your Gmail + all allowed user emails as Test Users
  • Create Credentials → OAuth Client ID → Web application
  • Authorized redirect URI: https://auth.yourdomain.com/source/oauth/callback/google/
  • Save Client ID and Client Secret

In Authentik admin → Directory → Sources → Create → OAuth2/OpenID Source:

Name:             Google
Slug:             google
Consumer Key:     [your Google Client ID]
Consumer Secret:  [your Google Client Secret]
OIDC Well-Known:  https://accounts.google.com/.well-known/openid-configuration
User matching:    Match by email (preferred)
☑ Allow users to change username

Now users see "Sign in with Google" on the Authentik login page. They click it, authenticate with Google, and land in Authentik.

Step 5: Expose Authentik via Cloudflare Tunnel 
# In your cloudflared config.yml, add:
ingress:
  - hostname: auth.yourdomain.com
    service: http://localhost:9000
  # ... other services
  - service: http_status:404

Then in Cloudflare Zero Trust → Access → Applications → Add Application for auth.yourdomain.com. Set a policy that only allows specific Gmail addresses. This is a second layer of protection before anyone even reaches Authentik.

Step 6: Configure Caddy forward_auth for all services

This is the magic that protects services that don't natively support OIDC. Every request to a service first hits Authentik for validation:

# In your Caddyfile — template for any protected service:
movies.yourdomain.com {
    forward_auth localhost:9000 {
        uri /outpost.goauthentik.io/auth/caddy
        copy_headers X-authentik-username X-authentik-groups X-authentik-email
        trusted_proxies private_ranges
    }
    reverse_proxy localhost:8096  # Jellyfin
}

# The X-authentik-username header is passed to the service
# so each service knows WHO is logged in.
# Services can use this header to route saves, history, etc.
💡
Services with native OIDC (Jellyfin, Open WebUI, Immich, Nextcloud, Vikunja) can use direct OIDC integration — this is slightly better UX. Services without native OIDC support (Navidrome, EmulatorJS save server, Kiwix) use Caddy forward_auth instead. Both approaches result in users being authenticated through Authentik.

Setting up Groups in Authentik
Create groups in Authentik admin

Directory → Groups → Create group for each: "authentik-admins", "authentik-family", "authentik-friends". Assign users to groups.

Pass group membership to services via OIDC claims

In Authentik → Applications → [each app] → Property Mappings → add a scope mapping that includes groups in the ID token:

# In Authentik: Customization → Property Mappings → Create
# Type: Scope Mapping
Name: groups
Scope Name: groups
Expression:
return [group.name for group in request.user.ak_groups.all()]
Nextcloud: map groups to storage quotas

When a user authenticates via Authentik OIDC, Nextcloud receives their group. Use the Nextcloud "User LDAP/OpenID Connect" app to map Authentik groups to Nextcloud storage quotas:

# In Nextcloud config.php (add via occ command):
php occ config:app:set user_oidc group_quota_mapping \
  --value '{"authentik-family":"50 GB","authentik-friends":"5 GB","authentik-admins":"0 B"}'
# "0 B" = unlimited in Nextcloud's quota system
💡
Individual overrides: In Nextcloud admin → Users → find the user → set a custom storage quota. This overrides the group default. So John can get 5GB while Dave gets 1GB, even if both are in the same Authentik group.

💾 Save File Migration Scripts
🐉
The problem: Before Authentik, services created user accounts with local usernames. After Authentik, the "username" passed to services comes from the Google account's email prefix (e.g. alice from alice@gmail.com). These migration scripts rename existing save/data directories to match the new Authentik-derived usernames.
🎮 Emulation Save Files Migration

Old saves are in D:\Emulation\saves\[old-username]\. New saves go in D:\Emulation\saves\[gmail-prefix]\.

# migrate_emu_saves.py
import os, shutil, json

# Map old local usernames → new Gmail prefixes
USERNAME_MAP = {
    'alice_local':  'alice',    # alice_local → alice@gmail.com prefix
    'bob123':       'bob',
    'charlie_mc':   'charlie',
}

SAVE_BASE = 'D:/Emulation/saves'
for old_name, new_name in USERNAME_MAP.items():
    old_path = os.path.join(SAVE_BASE, old_name)
    new_path = os.path.join(SAVE_BASE, new_name)
    if os.path.exists(old_path) and not os.path.exists(new_path):
        shutil.copytree(old_path, new_path)
        print(f'Migrated: {old_name} -> {new_name}')
    elif os.path.exists(old_path) and os.path.exists(new_path):
        print(f'CONFLICT: both {old_name} and {new_name} exist — merge manually')

print('Done. Verify saves work before deleting old directories.')

# Step 1: Before enabling Authentik OIDC in ABS,

General Migration Principle
💡
For every service: Before enabling Authentik OIDC, update all existing user accounts to use the user's exact Gmail address as their email/username. Most services match on email during OIDC login, so existing data is automatically preserved. For save-file-based systems (emulators), use the rename script above.
💣 Block & Nuke System
🐉
The power of Authentik: Because every service authenticates through Authentik, blocking or nuking a user in Authentik instantly removes their access from ALL services simultaneously — no need to log into each app individually.
🚫 Temporary Block (from Authentik admin)

Authentik admin → Directory → Users → click user → Deactivate toggle. The user's session tokens are revoked immediately. They can't log into any service. Re-activate whenever you want.

# Or via Authentik API (for automation):
curl -X PATCH https://auth.yourdomain.com/api/v3/core/users/[user-pk]/ \
  -H "Authorization: Bearer [authentik-api-token]" \
  -H "Content-Type: application/json" \
  -d '{"is_active": false}'
⏰ Timed Block (expires automatically)

Authentik doesn't have native timed blocks, but you can combine Authentik deactivation with a scheduled Task Scheduler re-activation:

# Block now:
# Authentik admin → deactivate user

# Schedule re-activation (Task Scheduler, one-time):
# Action: PowerShell script that calls Authentik API to re-activate
$headers = @{ "Authorization" = "Bearer YOUR_TOKEN"; "Content-Type" = "application/json" }
Invoke-RestMethod -Uri "https://auth.yourdomain.com/api/v3/core/users/[user-pk]/" `
  -Method Patch -Headers $headers -Body '{"is_active": true}'
💣 Full Nuke (permanent ban + save deletion)

⚠️ This is irreversible

A nuke does ALL of the following:

  • Delete user from Authentik (kills all service access instantly)
  • Block their IP in Windows Firewall
  • Delete D:\Emulation\saves\[username]\ (all game saves)
  • Delete their Nextcloud files (D:\NextcloudData\[username]\)
  • Create a Feedback ticket: "Remove [user] from Jellyfin and any manual accounts"
  • Services with Authentik OIDC automatically lose the user when the Authentik account is deleted
# nuke_user.py — run from server terminal
import subprocess, shutil, os, json, requests
from datetime import datetime

def nuke_user(username, gmail, authentik_token, authentik_base_url, user_pk):
    print(f"NUKING: {username} ({gmail})")

    # 1. Delete from Authentik (kills ALL service access immediately)
    r = requests.delete(
        f"{authentik_base_url}/api/v3/core/users/{user_pk}/",
        headers={"Authorization": f"Bearer {authentik_token}"}
    )
    print(f"  Authentik delete: {r.status_code}")

    # 2. Block IP in Windows Firewall (if you know their IP)
    # (IP is in the Authentik audit log: Directory → Audit Log → filter by user)
    # subprocess.run(['netsh','advfirewall','firewall','add','rule',
    #                f'name=NUKE_{username}','dir=in','action=block',f'remoteip={ip}'])

    # 3. Delete game saves
    save_path = f'D:/Emulation/saves/{username}'
    if os.path.exists(save_path):
        shutil.rmtree(save_path)
        print(f"  Deleted saves: {save_path}")

    # 4. Delete Nextcloud files
    nc_path = f'D:/NextcloudData/{username}'
    if os.path.exists(nc_path):
        shutil.rmtree(nc_path)
        print(f"  Deleted Nextcloud: {nc_path}")

    # 5. Create reminder ticket
    tickets = []
    try: tickets = json.load(open('D:/Server/Dashboard/tickets.json'))
    except: pass
    tickets.insert(0, {
        'id': f'TKT-{len(tickets)+1:04d}',
        'user': 'system',
        'subject': f'[NUKE] Remove {username} from manual accounts',
        'body': f'User {username} ({gmail}) was nuked on {datetime.now().strftime("%Y-%m-%d %H:%M")}.\n\nManual steps needed:\n- Remove from Jellyfin if local account exists\n- Remove from Calibre-Web if local account exists\n- Check Audiobookshelf for orphaned data',
        'status': 'unread',
        'date': datetime.now().strftime('%Y-%m-%d %H:%M'),
        'ip': 'system'
    })
    json.dump(tickets, open('D:/Server/Dashboard/tickets.json','w'), indent=2)
    print(f"  Created reminder ticket")
    print(f"NUKE COMPLETE for {username}")

# Usage:
# nuke_user('bob', 'bob@gmail.com', 'YOUR_TOKEN', 'https://auth.yourdomain.com', 42)
IP Block via Windows Firewall

Find the user's IP in Authentik: Directory → Audit Log → filter by username → look at login events for "Client IP".

# Block an IP permanently:
netsh advfirewall firewall add rule name="BLOCK_1.2.3.4" dir=in action=block remoteip=1.2.3.4

# Unblock:
netsh advfirewall firewall delete rule name="BLOCK_1.2.3.4"

Also add the IP to your Cloudflare Security → WAF → IP Access Rules → Block for a double-layer block at the CDN level before traffic even reaches your server.

🏆 Achievement System
🎮 RetroAchievements (emulated games)

Users connect their RetroAchievements.org account in their User Settings. EmulatorJS sends their RA credentials to retroachievements.org to unlock achievements during gameplay. Authentik provides the username — no separate RA login in the game UI needed if RA credentials are stored per-user in Authentik's user attributes.

# Store RA credentials in Authentik user attributes:
# Authentik admin → Directory → Users → [user] → Attributes:
{
  "ra_username": "AliceRetro",
  "ra_token": "abc123def456"
}

# EmulatorJS server reads these from the X-authentik-username header,
# then fetches user attributes via Authentik API:
GET /api/v3/core/users/?username=[username]
→ returns user.attributes including ra_username and ra_token
📚 Book/Audiobook Achievements

Audiobookshelf webhook fires on "Audiobook Finished" → posts to achievement server (port 9003) → server grants achievement for that Authentik user. Calibre-Web fires a webhook when user marks a book "Read."

# In ABS: Settings → Notifications → Webhooks
# Event: Audiobook Finished
# URL: http://localhost:9003/api/achievement/book
# Body:
{
