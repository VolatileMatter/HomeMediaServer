📁 Step 1 — Clone Repository

Sandspiel requires a build step — it is written in Rust and compiled to WebAssembly. The built output is what gets served.

cd D:\Server\BrowserGames
git clone https://github.com/maxbittker/sandspiel.git sandspiel
⚠️
Build required. The raw repo cannot be served directly. You need the compiled dist/ or build/ output. See Step 2.
⚙️ Step 2 — Build

Sandspiel uses Rust + wasm-pack. You need Rust and Node.js installed.

# Install wasm-pack if not already installed
cargo install wasm-pack

cd D:\Server\BrowserGames\sandspiel

# Build the WASM module
wasm-pack build --target web

# Install JS deps and bundle
npm install
npm run build
# Serve the www/ folder (or dist/ — check the repo's README for current output location)
💡
Alternatively, check the repo's Releases page on GitHub — a pre-built static version may be available and avoids the Rust toolchain requirement entirely.

🔐 Step 4 — Access Control

Sandspiel is served under the Browser Games hub. Access is restricted to Tailscale-enrolled devices only — the hub subdomain is not publicly reachable. Authentik forward_auth at the hub level gates all sub-paths including /sandspiel/.

No additional Caddy block or Authentik Application entry is needed for this game specifically. It inherits the hub's auth.

⚠️
Sandspiel's WASM bundle is large (~5–15 MB). Ensure Caddy is not applying gzip to .wasm files incorrectly — Content-Type: application/wasm must be set for browsers to accept it.
✅ Sandspiel is a fully client-side WASM toy — after the initial page load, it makes zero network requests. All physics runs locally in the browser.

🎮 What Is This?

Sandspiel is a falling-sand simulator — a genre of toy where you paint with different materials and watch them interact according to physical rules. Sand falls and piles, water flows and evaporates near fire, plant spreads and burns, fungus grows through moisture, and lava hardens into stone.

It was created by Max Bittker and is notable for being one of the first falling-sand toys built entirely in Rust compiled to WebAssembly, making it genuinely fast compared to JavaScript-based equivalents. The physics simulation runs at 60fps in modern browsers.

There is no goal and no way to lose. It is a creative sandbox — you paint, experiment, and see what happens. Some people build elaborate contraptions; others just enjoy watching fire spread through a field of plants.

📖 How to Play
  • Select an element from the palette on the right (sand, water, fire, plant, wall, etc.)
  • Click and drag on the canvas to place that element
  • Watch elements interact — fire burns plant, water extinguishes fire, sand piles up
  • Experiment with combinations: lava + water = stone; fire + plant = spread of flame
  • Use the brush size slider to place large or fine amounts
  • The canvas is finite — elements that fall off the bottom disappear
Controls

🧪 Element Reference
ElementBehaviour
🏖️ SandFalls, piles on surfaces and other sand. Displaced by water flowing beneath it.
💧 WaterFlows laterally and downward. Evaporates near fire. Fills enclosed spaces.
🔥 FireSpreads to flammable materials (plant, wood, oil). Burns out over time.
🌿 PlantGrows when near water. Burns when near fire. Static once dry.
🌋 LavaFlows slowly. Hardens into stone on contact with water. Sets fire to flammables.
⬛ WallSolid, immovable. Useful for building enclosures and channels.
🍄 FungusSpreads through moisture. Will consume plant and other organic materials.
💨 GasRises and disperses. Ignites explosively near fire.

The full element list is in the in-game palette. The above are the most commonly used.

⚠️ Known Limitations
IssueDetail
Browser supportRequires WebAssembly support. All modern browsers (Chrome, Firefox, Edge, Safari 14+) work. Very old browsers will not.
MobileTouch input works for drawing. The element palette is small on phone screens — tablet or desktop recommended.
PerformanceRuns well on mid-range hardware. Filling the entire canvas with active elements (fire spreading through dense plant) may cause minor frame drops on older machines.
No saveClosing or refreshing the tab clears the canvas. There is no save/load feature.
WASM load timeFirst load downloads ~5–15 MB of WASM. Subsequent loads use the browser cache — effectively instant.
