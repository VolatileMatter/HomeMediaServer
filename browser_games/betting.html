🐳 Step 1 — Clone & Configure
cd ~/services
git clone https://github.com/Satyam1616/1xBet-Clone.git betting
cd betting

# Copy env files — there may be separate ones for frontend/backend:
cp .env.example .env
# Edit all .env files thoroughly before proceeding
⚠️
Set strong values for all secrets in .env before starting. JWT_SECRET, POSTGRES_PASSWORD, and any session keys must not be left at defaults. Even on a private server, defaults are a bad habit.
🐳 Step 2 — Docker Compose

The repo ships a docker-compose.yml covering all 5 services. Review it, then start:

docker compose up -d --build

Expected running containers after a successful start:

ContainerRoleTypical port
frontendReact SPA3000
backend / apiNode.js REST API5000
postgresPrimary database5432 (internal)
redisSessions / caching6379 (internal)
nginxInternal reverse proxy80 or 8080

Point Caddy at the nginx container's exposed port — not the frontend directly.

# Run database migrations after containers start:
docker exec betting-api npx prisma migrate deploy
# or for Sequelize:
docker exec betting-api npx sequelize-cli db:migrate
🌐 Step 3 — Caddy Configuration
betting.{$DOMAIN} {
  forward_auth  {
    uri /outpost.goauthentik.io/auth/caddy
    copy_headers X-authentik-username X-authentik-groups
  }
  reverse_proxy localhost:NGINX_PORT
}

🎮 What Is This?

This is a full-stack clone of the 1xBet sports betting interface — one of the largest sports betting platforms globally. The clone replicates the UI and core mechanics: user registration, a virtual wallet, live (virtual) odds on sporting events, bet placement, a casino section with virtual games, and an admin panel for managing the platform.

The purpose here is as an elaborate entertainment platform for a friend group — running a private "betting league" with virtual stakes on real sporting events, or simply exploring what a full betting platform infrastructure looks like from the inside. The admin panel is a genuine feature, allowing the server owner to manage events, adjust virtual odds, and control the platform.

Architecturally it's the most complex application in the entire hub — five services working together, session management via Redis, and full relational data in PostgreSQL. Running it successfully is itself an accomplishment worth documenting.

⚠️ Known Limitations
IssueDetail
Highest setup complexity5 containers, multiple .env files, migrations, and nginx internal routing. Set aside a full evening for initial setup and debugging.
Live odds require managementVirtual events and odds must be manually created or seeded via the admin panel — there's no real sports data feed wired in by default.
Maintenance overhead5 services to keep healthy, updated, and backed up. More ongoing maintenance than any other game in the hub.
KYC in private useKYC verification is a UI flow — for a private server among trusted friends, leave verification requirements off in the admin panel.
