Game files live under D:\Server\BrowserGames\GAMENAME\.

npm run build
# Output will be in dist\ — serve that folder, not the source root
💡
The Browser Games hub serves all games from a single Python http.server (or nginx) root. The hub's games.json registry tells the hub what games exist — always update it.
📋 Step 2 — Register in games.json

Add an entry to D:\Server\BrowserGames\games.json. This is the single source of truth for the hub's game list. The hub reads this file at load time to populate the lobby. 

{
  "id":          "GAMENAME",                    // unique slug, lowercase, no spaces
  "name":        "GAME NAME",                   // display name
  "icon":        "🎮",                          // emoji icon shown in the lobby
  "description": "One-line description.",       // shown in the game card
  "folder":      "GAMENAME",                    // subfolder name under BrowserGames\
  "entry":       "index.html",                  // entry point within the folder
  "category":    "singleplayer",               // "singleplayer" | "multiplayer"
  "highScores":  true,                          // true | false
  "players":     "1",                           // "1" | "2" | "2-4" | "1-8" etc.
  "tags":        ["puzzle", "arcade"]           // optional tags for filtering
}
⚠️
Always restart the http.server after editing games.json — it doesn't hot-reload. Use nssm restart BrowserGamesServer (or whatever the service name is).
🏆 Step 3 — High Score API Integration IF APPLICABLE

The Browser Games hub has a shared high score API running alongside it (a small Python/Flask server). Games submit scores via a simple POST call. If this game supports high scores, wire it up here. 

// In the game's JS, when the player achieves a score:
fetch('/api/scores/submit', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    game:  'GAMENAME',        // must match games.json id
    score: playerScore,       // integer or float — define the unit in the High Score tab
    meta:  { level: 3 }       // optional — any game-specific context
  })
});

// The hub reads the username from X-authentik-username header automatically.
// The game's JS does NOT need to supply the username.
🐉
Do not modify the game's source to make API calls if you don't control the source. In that case, use the hub's score proxy: host a lightweight wrapper page that launches the game in an iframe and listens for a postMessage score event. Document the wrapper approach in the High Score tab.
🔐 Step 4 — SSO & Reverse Proxy

Browser games are served under the games. subdomain, behind the hub's Authentik forward_auth block. Individual game pages do not need their own Caddy block — they're all sub-paths of the hub. 

# No new Caddy block needed for individual games.
# The hub's existing block handles all paths under /games/.
#
# If this game requires special headers, cookies, or CORS,
# add a path-specific override inside the hub's Caddy block:
games.{$DOMAIN} {
  # ... existing forward_auth block ...

  # Game-specific path override example:
  route /GAMENAME/* {
    header Access-Control-Allow-Origin "*"   # if the game loads workers
  }
}

The only time a new Caddy block is needed is if the game has a separate backend process (e.g. a WebSocket multiplayer server). In that case, proxy that backend to a dedicated subdomain or sub-path and document it in the Multiplayer tab.

🛡️ Step 5 — Firewall

Static files are served by the hub's existing process — no new firewall rule needed unless this game runs an additional backend on a new port. 

# Only needed if the game has a separate backend (e.g. WebSocket server):
netsh advfirewall firewall add rule ^
  name="BrowserGames - GAMENAME Backend" ^
  dir=in action=allow protocol=TCP localport=BACKEND_PORT
