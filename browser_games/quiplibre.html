📁 Step 1 — File Placement

Quiplibre is a single HTML file. The entry point is quiplibre.html (not index.html), so you'll need a redirect shim.

# Clone the repo:
cd D:\Server\BrowserGames
git clone https://github.com/wyattscarpenter/quiplibre.git quiplibre

# The game entry point is quiplibre.html, not index.html.
# The repo ships an index.html that already redirects to quiplibre.html,
# so serving the folder root should work correctly out of the box.
💡
The repo includes a symlink index.html → quiplibre.html. On Windows, git may check this out as a text file containing the symlink target path rather than an actual redirect. If index.html doesn't redirect correctly, replace it with a one-line meta-refresh: <meta http-equiv="refresh" content="0;url=quiplibre.html">

🔐 Step 3 — Access Control

Served under the Browser Games hub. Access requires both Tailscale enrollment and a valid Authentik SSO session. No additional Caddy block needed — inherits the hub's forward_auth.

⚠️
Quiplibre requires an internet connection at play time to reach the PeerJS brokering server (unpkg.com) and Google's STUN server. It will not function in a fully offline or internet-isolated environment, even on Tailscale. All players must have internet access, not just server-side LAN access.

🎮 What Is This?

Quiplibre is a free, open-source implementation of Quiplash — the party game where everyone answers the same absurd prompt and then votes on who gave the best answer. No one is right or wrong; you're playing to be funny.

Each player joins a shared session using a room code. Everyone is dealt two prompts to answer. Answers are then shown head-to-head and all players vote on their favourite. The crowd's reaction is the whole game — no points system, no leaderboard, just vibes.

Unlike Jackbox's Quiplash, Quiplibre requires no TV screen, no app purchase, and no subscription. Each player opens the same URL and either hosts or joins. It works across the room on a LAN or remotely across the Tailnet, as long as everyone has internet access for the PeerJS broker.

▶ How to Play

One person hosts, everyone else joins.

1. Host opens the URL
→
2. Click "Host a new game"
→
3. Share the room code
4. Players open URL + "Join a game"
→
5. Enter room code + name
→
6. Host starts the game

Each player gets two prompts. Type your funniest answer. When everyone's submitted, answers go head-to-head and players vote. Repeat until prompts run out — or until someone gets too competitive about a game with no score.

💡
There is no "big screen" mode — each player sees everything on their own device. For a more Jackbox-like experience, have one player share their screen over Discord/video call while everyone else plays on their own phone or laptop.
🃏 Custom Prompts

The prompts list is editable directly in quiplibre.html. The array is called prompts and lives near the top of the <script> block. Format:

const prompts = [
  "A terrible name for a hospital",
  "The worst thing to say at a job interview",
  // ← Add your prompts here, with trailing commas
  "Your custom prompt goes here",
];

Use single or double quotes. Don't forget the comma after each entry. The game draws prompts randomly each session — adding more prompts just expands the pool.

🏗️ How Multiplayer Works

Quiplibre uses PeerJS, a WebRTC wrapper library loaded from unpkg.com at runtime. The host's browser opens a peer connection through PeerJS's public brokering server; joining players connect to the host's peer ID (the room code) directly. All game state is transmitted peer-to-peer — no server stores any game data.

STUN server used: stun.l.google.com:19302 (Google's public STUN). This is hardcoded in the game and requires internet access.

⚠️
Because connections are brokered through PeerJS's public infrastructure (unpkg.com, Google STUN), Quiplibre cannot be played in a fully air-gapped or internet-isolated network. Tailscale handles access control and routing to the server; the PeerJS WebRTC connections are made directly between players' browsers and require their own internet access.

There is no backend server component to manage. No additional Docker container, no extra Caddy route, no firewall port — the hub's existing static file serving is all that's needed.

🔀 Matchup Algorithm

Each player answers two prompts. Answers are matched head-to-head for voting using the "Carpenter shuffle" — a left-shifted array followed by randomised swaps that preserve the no-self-match constraint. With fewer than 4 players the shuffle has no valid swap candidates and falls back to the shifted-only order, so matchups will be predictable but the game still functions.

⚠️ Known Issues
IssueDetail
Safari / WebKitWebRTC does not work. All iOS browsers are affected due to WebKit enforcement. Players must use Chrome or Firefox.
PeerJS dependencyIf unpkg.com or PeerJS's brokering infrastructure is down, hosting/joining will fail. No offline fallback exists.
Host disconnectsThe host's browser is the session authority. If the host closes their tab, the game ends for all players.
No reconnectIf a player's connection drops mid-game, there is no rejoin mechanism — they must start fresh.
