⚠️ Architecture Note
⚠️
BrowserQuest is not a static file game. It requires a persistent Node.js WebSocket server for multiplayer. Unlike other browser games in this hub, it needs its own Docker container. The client HTML/JS is served statically, but the game itself is non-functional without the game server.
📁 Step 1 — Clone Repository
cd ~/services
git clone https://github.com/mozilla/BrowserQuest.git browserquest
cd browserquest
🐳 Step 2 — Docker Compose

Create ~/services/browserquest/docker-compose.yml:

services:
  browserquest:
    build: .
    container_name: browserquest
    restart: unless-stopped
    ports:
      - "8800:8000"   # WebSocket game server
      - "8801:80"     # Static client files
    environment:
      - NODE_ENV=production

Then build and start:

docker compose up -d --build
⚠️
The original Mozilla repo targets an older Node.js version. If the build fails, check the repo's README for the required Node version. Community forks (see Source tab) may have updated dependency support.
🌐 Step 3 — Caddy Configuration

Add to :

browserquest.{$DOMAIN} {
  forward_auth  {
    uri /outpost.goauthentik.io/auth/caddy
    copy_headers X-authentik-username X-authentik-groups
  }

  # WebSocket proxy for the game server
  handle /ws* {
    reverse_proxy localhost:8800
  }

  # Static client files
  handle {
    reverse_proxy localhost:8801
  }
}
💡
BrowserQuest uses WebSockets for all game communication. Caddy handles WebSocket upgrades automatically — no special configuration needed beyond the reverse_proxy directive.
📋 Step 4 — Register in index.json

BrowserQuest is best registered as an external URL entry since it runs on its own subdomain, not as a sub-path of the games hub:

✅ BrowserQuest requires both the client (static files) and the game server (Node.js/WebSockets) to be running. Check Docker if the game connects but the world is empty.

🎮 What Is This?

BrowserQuest was released by Mozilla in 2012 as a showcase for HTML5 capabilities — specifically WebSockets, Canvas, LocalStorage, and Web Audio. It demonstrated that a real-time multiplayer game could run entirely in a browser without Flash or plugins, which was a meaningful statement at the time.

The game itself is a short top-down RPG with tile-based movement. Players share a world and can see each other move in real time. There are enemies to fight, items to collect (sword, armour, belt upgrades), and a final boss to defeat. The full adventure takes about 15–30 minutes. It's not deep, but it was never meant to be — it's a technology demo that happens to be genuinely fun.

The Mozilla-hosted version was shut down, but the open-source repository remains available for self-hosting. Running it on this server means your friend group gets a private shared world.

📖 How to Play
  • Enter a name when prompted — you spawn in the starting area
  • Click to move; click enemies to attack them
  • Pick up dropped items to upgrade your gear
  • Other players appear in your world in real time — you can see their movements and fights
  • Follow the map to progress; there's a dungeon and a final boss at the end
  • Death respawns you at the start — you lose your items

⚠️ Known Limitations
IssueDetail
No save systemDisconnecting loses your progress and items. The game is short enough to complete in one sitting.
Shared world stateAll players are in the same world. Enemy kills and item drops are visible to all. No instancing.
Server dependencyThe game is unplayable without the Node.js game server. If multiplayer breaks, check Docker first.
Old Node.js targetOriginal code targets Node 0.8. Community forks update this — see Source tab for modern alternatives.
👥 Multiplayer is the entire point of BrowserQuest. The game has no meaningful single-player mode.
🌐 Server Architecture
ComponentDetail
ProtocolWebSockets (binary messages)
BackendNode.js game server managing world state, entity positions, combat
World modelSingle shared world — all connected players coexist
Player limitConfigurable in server config; default supports 20–50 concurrent players comfortably
PersistenceNone — world state is in-memory only. Server restart resets everything.
🔧 Server Management
# Check game server status:
docker compose -f ~/services/browserquest/docker-compose.yml ps

# View live logs:
docker compose -f ~/services/browserquest/docker-compose.yml logs -f

# Restart if connections are dropping:
docker compose -f ~/services/browserquest/docker-compose.yml restart
⚠️ Known Issues
IssueDetail
WebSocket path conflictIf serving behind Caddy, ensure the /ws path routes to port 8800 (game server), not 8801 (static files).
Player ghostingIf a player disconnects uncleanly their character may linger briefly. Resolves on its own within ~30 seconds.
Concurrent playersSmall friend-group usage (2–8 players) is well within the server's capacity.
