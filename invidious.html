Invidious

// YouTube Frontend · Port · Docker WSL2 · Authentik forward_auth //

A privacy-respecting, self-hosted YouTube frontend. No Google tracking, no ads, no Shorts push. Per-user subscription feeds backed by a Postgres database, SponsorBlock built in, and uBlock-style content filtering via the proxy — all locked behind Authentik so only server members can access it. Works on desktop and mobile browsers with no app install required.

⚙️ Step 1 — Confirm WSL2 + Docker WSL2 + Docker

Invidious runs in Docker on WSL2 (same pattern as Immich and Authentik). If either of those are already running, WSL2 and Docker are ready — skip to Step 2.

🔐 Verify Docker is available inside WSL2 
wsl -d Ubuntu -- docker ps

If you see a table header (even empty), Docker is ready. If you get a connection error, start Docker: open the Ubuntu WSL2 terminal and run sudo service docker start.

⚙️ Step 2 — Deploy Invidious Docker Compose
🔐 Create the directory and fetch the official compose file 
mkdir ~/invidious && cd ~/invidious

# Grab the official Docker Compose from the Invidious repo:
wget -O docker-compose.yml \
  https://raw.githubusercontent.com/iv-org/invidious/master/docker-compose.yml
🔐 Create the Invidious config file

Create ~/invidious/config/config.yml (the directory must exist first):

mkdir -p ~/invidious/config

cat > ~/invidious/config/config.yml << 'EOF'
# ── Database ─────────────────────────────────────────────────
db:
  user: kemal
  password: CHANGE_THIS_STRONG_PASSWORD
  host: invidious-db
  port: 5432
  dbname: invidious

# ── Core settings ────────────────────────────────────────────
check_tables: true
external_port: 443
domain: invidious.
https_only: true

# ── Authentication / Registration ────────────────────────────
# Registration is disabled — accounts are created by the admin only.
# Users authenticate via Authentik forward_auth at the Caddy layer;
# their Invidious account is a thin record that stores subscriptions.
registration_enabled: false

# ── Homepage defaults ─────────────────────────────────────────
# Every user's homepage shows their subscription feed, not trending.
default_home: Subscriptions
feed_menu:
  - Subscriptions
  - Playlists
  - History

# ── Privacy / Proxy ──────────────────────────────────────────
# Route all media through the Invidious server so the client
# never contacts Google. This is what makes adblock work:
# no direct connection to googlevideo.com, no ad requests land.
invidious_companion:
  - private_url: http://invidious-companion:8282

# ── SponsorBlock ─────────────────────────────────────────────
# Built-in SponsorBlock integration — users can set their
# skip preferences in account settings.
sponsorblock_server: https://sponsor.ajay.app

# ── Cosmetic defaults ─────────────────────────────────────────
popular_enabled: false
statistics_enabled: false
EOF
⚠️
Replace CHANGE_THIS_STRONG_PASSWORD with a real password. It must match the POSTGRES_PASSWORD you set in the environment below.
🔐 Edit the Docker Compose environment

Open ~/invidious/docker-compose.yml and set the Postgres password to match your config. The official compose file has a placeholder — find the POSTGRES_PASSWORD line under the invidious-db service and set it to the same value as above.

# Key environment vars to verify / set in docker-compose.yml:
# Under service: invidious-db
#   POSTGRES_PASSWORD: CHANGE_THIS_STRONG_PASSWORD   ← match config.yml
#   POSTGRES_USER: kemal
#   POSTGRES_DB: invidious
#
# Under service: invidious
#   INVIDIOUS_CONFIG_FILE: /invidious/config/config.yml
🔐 Add the config volume mount to the compose file

The official compose file may not mount your config folder by default. Under the invidious service's volumes: section, ensure this line is present:

    volumes:
      - ./config/config.yml:/invidious/config/config.yml:ro
🔐 Start Invidious 
cd ~/invidious
docker compose up -d

# Watch startup logs — first run pulls images (~500 MB) and initialises the DB:
docker compose logs -f invidious

Ready when you see Invidious is ready to serve in the logs. Access it locally at http://localhost:.

🔐 Step 3 — Create Invidious Accounts for Each User

Because registration_enabled: false is set in the config, users cannot self-register. The admin must create one Invidious account per server user, using a username that exactly matches their Authentik username. This is the link between SSO identity and their subscription database row.

💡
The Authentik forward_auth layer authenticates the user, but Invidious uses its own internal session once they land on the page. Matching usernames means you can later auto-login them via a proxy header script, or just have them log in once and stay logged in forever via cookie.
🔐 Temporarily enable registration to create accounts

Edit ~/invidious/config/config.yml and change:

registration_enabled: true

Then restart Invidious: docker compose restart invidious

🔐 Create each user's account

Go to http://localhost:/register and create an account for each user. Use their Authentik username as the Invidious username. Password can be anything — users won't need it if they stay logged in via cookie, but record it somewhere secure (Vaultwarden).

🔐 Disable registration again 
# In ~/invidious/config/config.yml:
registration_enabled: false

docker compose restart invidious
🔐 Step 4 — Caddy Reverse Proxy + Authentik SSO → SSO TAB
Auth pattern: Caddy forward_auth

Invidious has no native OIDC support. Caddy validates the Authentik session on every request before the request ever reaches Invidious. If you're not logged into Authentik, you get the Authentik login page — Invidious is completely invisible to unauthenticated users.

🔐 Create the Authentik application (Proxy Provider — forward auth)

In Authentik Admin → Applications → Providers → Create → Proxy Provider:

Name:            Invidious
Authorization:   (your default authorization flow)
Forward auth:    Single Application
External Host:   https://invidious.

Then create the Application:

Authentik Admin → Applications → Applications → Create
Name:       Invidious
Slug:       invidious
Provider:   (the Proxy Provider above)
Launch URL: https://invidious.
Icon:       (optional — use a YouTube/Invidious logo)

In the Application's Policy/Binding tab, bind your server user group so only your users can access it. Anyone not in that group gets denied at Authentik — Invidious never sees the request.

🔐 Add to your Caddyfile

Open and add:

invidious. {
    forward_auth localhost:9000 {
        uri /outpost.goauthentik.io/auth/caddy
        copy_headers X-authentik-username X-authentik-groups X-authentik-email
        trusted_proxies private_ranges
    }
    reverse_proxy localhost:
}

Reload Caddy:

caddy reload --config ""
🔐 Test the SSO gate

Open an incognito window and navigate to https://invidious.. You should be immediately redirected to the Authentik login page. Log in with a valid server account — you should land on the Invidious subscription feed. Try with a non-member Google account — it should be denied by Authentik.

🛡️ Step 5 — Ad Blocking & SponsorBlock
How ad blocking works in Invidious

Invidious proxies all video and media through your server — the client never makes a direct request to googlevideo.com or any Google ad infrastructure. Because the media proxy strips the connection to Google's ad servers entirely, ads never load. There's nothing extra to install — it's a consequence of how Invidious works when the companion proxy is enabled (which the config above does).

SponsorBlock — per-user configuration

SponsorBlock is enabled server-side via the sponsorblock_server config key. Each user controls what gets skipped in their account preferences:

  1. Log into Invidious → click your username (top right) → Preferences
  2. Scroll to the SponsorBlock section
  3. Enable the segment types you want auto-skipped: Sponsor, Intro, Outro, Self-Promotion, Interaction Reminder, etc.
  4. Save — SponsorBlock applies to every video you watch from that point on
💡
Mobile tip: On mobile browsers, users can add the Invidious URL to their home screen (iOS: Share → Add to Home Screen; Android: browser menu → Add to Home Screen). It behaves like a PWA — full-screen playback, no browser chrome in the way.
📥 Step 6 — Import YouTube Subscriptions Per User
Method A — Import via Google Takeout (recommended)

This is the cleanest method. It exports your YouTube data directly from Google and imports it into Invidious in one click.

Export from Google Takeout
  1. Go to takeout.google.com
  2. Click Deselect all, then scroll down and enable only YouTube and YouTube Music
  3. Click Next step → choose Export once, .zip format
  4. Click Create export — Google will email you a download link (usually within a few minutes)
  5. Download and unzip the archive
  6. Navigate to: Takeout / YouTube and YouTube Music / subscriptions /
  7. You'll find a file named subscriptions.csv
Import into Invidious
  1. Log into Invidious at
  2. Click your username → Import/Export data
  3. Under Import YouTube subscriptions, click Choose file
  4. Select your subscriptions.csv file
  5. Click Import — all your subscriptions appear immediately in your feed
Method B — Subscribe manually inside Invidious

For users who don't want to go through Takeout, or who only want to subscribe to a few channels:

  1. Search for a channel by name in the Invidious search bar
  2. Click through to the channel page
  3. Click the Subscribe button (top of channel page)
  4. Repeat for each channel. Subscriptions are stored in the Postgres DB immediately.
Method C — Import from an existing Invidious or NewPipe export

If a user already has an Invidious account on another instance, or uses NewPipe/FreeTube on their phone, those apps can export a subscription list in Invidious-compatible JSON format. Use the same Import/Export data page and choose the JSON import option instead of CSV.

🗄️ Database Backup

The Postgres database holds all user subscriptions, preferences, watch history, and playlists. Back it up regularly — it's small (<50 MB typically) but irreplaceable.

🔐 Dump the database 
cd ~/invidious
docker compose exec invidious-db pg_dump -U kemal invidious \
  > ~/backups/invidious_$(date +%Y-%m-%d).sql
🔐 Restore from backup (if needed) 
cd ~/invidious
docker compose exec -T invidious-db psql -U kemal invidious \
  < ~/backups/invidious_YYYY-MM-DD.sql

⚠️ Known Breakpoints
  • YouTube extraction breaks regularly: Invidious reverse-engineers YouTube's private API. Google changes it constantly. When video playback stops working entirely, pull the latest Docker image first — that's almost always the fix.
  • Rate limiting (429 errors): If your instance gets heavy use, Google will throttle the server IP. Mitigation: the companion proxy helps spread load. If it persists, check the Invidious GitHub issues for workarounds.
  • Shorts often don't work: YouTube Shorts use a different internal API and Invidious support for them is inconsistent. This is expected — direct users to the regular video link instead.
  • Live streams are unreliable: Live stream support depends on Google's HLS endpoint being accessible. It works intermittently — don't rely on it for live content.
  • Proxy increases server bandwidth: Every byte of video goes through your server. If multiple users are watching 1080p simultaneously, bandwidth usage is real. Monitor with docker stats.

🛠️ Common Issues
Videos won't play / black screen on all videos
YouTube broke the extraction API — this happens every few weeks. Fix: cd ~/invidious && docker compose pull && docker compose up -d. If the latest image doesn't fix it, check github.com/iv-org/invidious/issues for an active issue thread — the community usually has a workaround or patched image within hours.
Getting redirected to Authentik login even when already logged in
Your Authentik session may have expired. Log into Authentik at directly and confirm you have an active session. If the session is valid but Caddy still redirects you, check that the Authentik outpost is running: docker compose -f ~/authentik/docker-compose.yml logs outpost.
Subscription feed is empty after logging in
The Invidious account exists but has no subscriptions yet. Import them via the Google Takeout CSV method (see First Time Setup → Step 6), or subscribe to channels manually. Make sure you're logged in — the feed only shows when authenticated.
SponsorBlock segments aren't being skipped
Check Preferences → SponsorBlock — the categories must be individually enabled (they don't default to on for existing accounts). If they are enabled and still not skipping, check that sponsorblock_server: https://sponsor.ajay.app is in your config.yml and that the server can reach that URL: docker compose exec invidious curl -s https://sponsor.ajay.app/api/skipSegments?videoID=dQw4w9WgXcQ | head -c 200.
Ads are still appearing
If you're seeing ads, the video proxy is not active. Check Preferences → Player → Proxy videos is set to Enabled. If proxy is on and ads still appear, something is wrong with the companion container — check its logs: docker compose logs invidious-companion. The companion container must be running for the proxy to work.
429 Too Many Requests errors in logs
Google is rate-limiting your server's IP. This usually clears within a few hours. If it's persistent, consider reducing concurrent users or checking if something is hammering the subscription feed refresh. The Invidious team tracks this in their issues — search for "429" to see current mitigation strategies.
How do I add a new user after initial setup?
Temporarily set registration_enabled: true in config.yml, restart Invidious (docker compose restart invidious), create their account at /register using their Authentik username, then set registration_enabled: false and restart again. Also add them to the Authentik group binding on the Invidious application so they can pass the SSO gate.
How do I check what's using bandwidth / if proxy is working?
Run docker stats in WSL2 while someone watches a video. The invidious container's network I/O should be climbing — that means video is flowing through your server. If network I/O is near zero while video plays, the proxy is not active.
