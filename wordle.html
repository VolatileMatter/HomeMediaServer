Wordle

// Custom Family Wordle · Port · Docker WSL2 · Authentik forward-auth //

A self-hosted fork of the classic Wordle game with a custom daily word list and SSO-integrated score tracking. Each day's word can be hand-curated via customWords.ts; when the custom list runs out it falls back to the standard word list. Scores are posted to a small API endpoint and stored in scores.json per user, with the authenticated username injected by the reverse proxy via X-Authentik-Username.

1 · Build & run the Docker image

The repo lives at userdata/common/wordle on the server. Build once; rebuild whenever customWords.ts or wordlist.ts change (they are baked into the image at build time).

Build the image 
cd userdata/common/wordle
docker build -t custom-family-wordle .
Start with Docker Compose 
docker compose up -d

The container listens on port 80 internally and is exposed on the host at port . The reverse proxy handles TLS and SSO.

Verify it's running 
docker ps | grep wordle
# Should show: wordle   Up   0.0.0.0:8081->80/tcp
2 · Score tracking API

The front-end calls POST /wordle-score (or your chosen endpoint) with a JSON body containing { score, status, date }. A small Node script reads the X-Authentik-Username header injected by Caddy and appends the result to userdata/common/wordle/scores.json.

⚠️
The score endpoint must run separately from the static game container. It needs to read the X-Authentik-Username header, so it must be behind the same Caddy forward-auth middleware. A lightweight node / express process on a separate port works; or add a second service to docker-compose.yml.
Install Node dependencies (if running natively) 
cd userdata/common/wordle
npm install express
Start the score server 
node score-server.js
# Ensure it listens on a port forwarded through Caddy
# and that Caddy injects X-Authentik-Username
3 · 🔐 SSO — Authentik forward-auth (admin step)

Wordle uses Caddy forward-auth via Authentik. There is no native OIDC support in the static app — Authentik gates the entire subdomain and the proxy injects the username header.

Create an Authentik application + provider

In the Authentik admin panel ():

  • Create a Proxy Provider (forward-auth, single application)
  • External host:
  • Internal host: http://localhost:
Add forward-auth to Caddyfile 
 {
    forward_auth localhost:9000 {
        uri /outpost.goauthentik.io/auth/caddy
        copy_headers X-Authentik-Username Remote-User
    }
    reverse_proxy localhost:
}

The X-Authentik-Username (aliased as Remote-User) header is what the score API reads to attribute scores to the logged-in user.

Reload Caddy and verify 
caddy reload --config

Navigate to — you should be redirected to Authentik before seeing the game.

4 · Customise the word list

Edit src/constants/customWords.ts in the repo to add or change curated daily words. The game's getSolution() function picks words by the day-index: custom words first, then the standard wordlist.ts once the custom list is exhausted.

💡
After editing customWords.ts, rebuild the Docker image — the word lists are compiled into the static bundle at build time. 
docker build -t custom-family-wordle .
docker compose up -d

🔑 How score attribution works

When a game ends the client fires a POST to the score endpoint. The reverse proxy has already authenticated the user and stamped the request with X-Authentik-Username. The score server reads that header and writes an entry to scores.json — no separate login or token needed inside the game itself.

⚠️ Potential Breakpoints
🔴 Custom word list exhausted
Once all entries in customWords.ts are used, the game silently falls back to the standard word list. This is fine functionally, but players may notice unfamiliar or harder words. Add new words and rebuild before the list runs out to keep the curated experience.
🟡 Score API goes down independently of the game
The static game container and the score server are separate processes. If the score server crashes, players can still play but scores won't be recorded. Monitor with docker ps and restart as needed. Consider adding the score server to docker-compose.yml so it restarts automatically.
🟡 Authentik outpost update breaks forward-auth headers
After Authentik updates, verify that X-Authentik-Username is still being forwarded correctly. The score server will silently write entries with a blank or wrong username if the header disappears. Check scores.json after an Authentik update to confirm usernames look right.

🛠️ Common Issues
❌ Game loads blank / white screen
Usually a failed build. Check the container logs: 
docker logs wordle
If you see JS errors, rebuild the image — a broken customWords.ts export (e.g. a word longer than 5 letters) can crash the bundle.
❌ Authentik login loop / 401 on the game URL
Check the Authentik outpost is running and reachable from Caddy. Also verify the Caddyfile forward_auth block points to the correct outpost address. Reload Caddy after any config change: 
caddy reload --config
🟡 Scores not being recorded
Check the score server is running (docker ps or node process). Confirm Caddy is forwarding X-Authentik-Username to the score endpoint — add a temporary debug log to the score server to print all incoming headers. Also verify the endpoint URL in the game's reportScore() function matches what Caddy is routing.
🟡 Wrong or missing username in scores.json
The X-Authentik-Username header is not reaching the score server. Check the Caddyfile copy_headers directive includes it, and that the score endpoint is behind the same forward_auth block as the game.
🟢 Word repeats / same word two days in a row
The day-index is calculated from a fixed epoch (check src/lib/words.ts). If the server clock drifted or was reset, the index may jump. Verify the container's timezone is set correctly in docker-compose.yml: 
environment:
  - TZ=Australia/Sydney   # adjust to your timezone
