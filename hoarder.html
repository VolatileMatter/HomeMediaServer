⚙️ Step 1 — Pre-flight Checks WSL2 + Docker
What you need before starting
  • WSL2 (Ubuntu) running with Docker — confirm with wsl -d Ubuntu -- docker ps
  • Ollama already running on your server at port — Hoarder will call it for AI tagging
  • Port free on the host
  • Two secrets generated (do this now, store in your password manager):
# Generate HOARDER_SECRET (NextAuth secret):
openssl rand -base64 36

# Generate MEILI_KEY (Meilisearch master key):
openssl rand -base64 36
⚠️
Generate both secrets before writing the compose file. If you change either after first run, you will need to wipe and recreate the Meilisearch index and all existing sessions.
⚙️ Step 2 — Deploy with Docker Compose Docker Compose
🔐 Create the directory inside WSL2 
mkdir -p ~/hoarder
cd ~/hoarder
🔐 Create docker-compose.yml

Create ~/hoarder/docker-compose.yml with the following content. Replace the placeholder values with the secrets you generated in Step 1.

services:
  web:
    image: ghcr.io/hoarder-app/hoarder:latest
    restart: unless-stopped
    volumes:
      - ./data/hoarder:/data
    ports:
      - ":3000"
    environment:
      - NEXTAUTH_URL=
      - NEXTAUTH_SECRET=PASTE_HOARDER_SECRET_HERE
      - MEILI_ADDR=http://meilisearch:7700
      - MEILI_MASTER_KEY=PASTE_MEILI_KEY_HERE
      - BROWSER_WEB_URL=http://chrome:9222
      - OLLAMA_BASE_URL=http://[server-lan-ip]:
      - INFERENCE_TEXT_MODEL=llama3.1
      - INFERENCE_IMAGE_MODEL=llava
      # Authentik OIDC — fill in after creating the Authentik application (Step 4)
      - OAUTH_WELLKNOWN_URL=.well-known/openid-configuration
      - OAUTH_CLIENT_ID=PASTE_CLIENT_ID_HERE
      - OAUTH_CLIENT_SECRET=PASTE_CLIENT_SECRET_HERE
      - OAUTH_PROVIDER_NAME=Authentik
    depends_on:
      - meilisearch
      - chrome

  meilisearch:
    image: getmeili/meilisearch:v1.11
    restart: unless-stopped
    volumes:
      - ./data/hoarder_meili:/meili_data
    environment:
      - MEILI_MASTER_KEY=PASTE_MEILI_KEY_HERE
      - MEILI_NO_ANALYTICS=true

  chrome:
    image: gcr.io/zenika-hub/alpine-chrome:123
    restart: unless-stopped
    command: [
      "--headless",
      "--disable-gpu",
      "--disable-software-rasterizer",
      "--remote-debugging-address=0.0.0.0",
      "--remote-debugging-port=9222",
      "--hide-scrollbars"
    ]
🔐 Start the stack 
cd ~/hoarder
docker compose up -d

# Verify all three containers are running:
docker compose ps

You should see web, meilisearch, and chrome all in state running.

🔐 Confirm Hoarder is reachable locally 
# From the WSL2 terminal:
curl -s -o /dev/null -w "%{http_code}" http://localhost:
# Expected: 200 or 302

Or open http://[server-lan-ip]: in a browser — you should see the Hoarder login page.

🔐 Open firewall port 
netsh advfirewall firewall add rule name="Hoarder" ^
  dir=in action=allow protocol=TCP localport=
⚙️ Step 3 — Caddy Reverse Proxy
🔐 Add the Hoarder block to your Caddyfile

Open and add:

hoarder. {
  reverse_proxy localhost:
}

Hoarder handles its own authentication via OIDC, so no forward_auth wrapper is needed here — Caddy only needs to proxy the traffic.

🔐 Reload Caddy 
caddy reload --config

Confirm loads (will redirect to login — that's expected until OIDC is configured).

⚙️ Step 4 — Authentik OIDC SSO · OIDC
💡
Hoarder supports OAuth2/OIDC natively — no plugin or forward_auth wrapper needed. You configure the provider via environment variables and Hoarder handles the login flow itself.
🔐 Create an OAuth2/OIDC provider in Authentik

Go to → Applications → Providers → Create → OAuth2/OpenID Provider.

  • Name: Hoarder
  • Client type: Confidential
  • Redirect URIs: /api/auth/callback/custom
  • Signing key: Select your default signing certificate
  • Scopes: openid, email, profile

Save the provider and note the Client ID and Client Secret.

🔐 Create the Authentik Application

Go to Applications → Applications → Create.

  • Name: Hoarder
  • Slug: hoarder
  • Provider: Select the provider you just created
  • Launch URL:
🔐 Update the compose file with your OIDC credentials

Edit ~/hoarder/docker-compose.yml and fill in the three OAuth environment variables:

- OAUTH_CLIENT_ID=YOUR_CLIENT_ID_FROM_AUTHENTIK
- OAUTH_CLIENT_SECRET=YOUR_CLIENT_SECRET_FROM_AUTHENTIK
- OAUTH_PROVIDER_NAME=Authentik

The OAUTH_WELLKNOWN_URL should already be set from Step 2. Verify it matches the Authentik slug hoarder:

- OAUTH_WELLKNOWN_URL=.well-known/openid-configuration
🔐 Restart the web container to pick up the new env vars 
cd ~/hoarder
docker compose up -d --force-recreate web
Test SSO login

Open in an incognito window. You should be redirected to Authentik to log in, then land in Hoarder as an authenticated user.

Verify the Caddyfile entry

Confirm the reverse proxy block for Hoarder is in place at and that Caddy has reloaded:

caddy reload --config

🤖 How AI Tagging Works

When you save a bookmark, Hoarder sends the page content to your local Ollama instance (llama3.1) and asks it to generate tags and a summary. For images and screenshots, it uses llava to describe the visual content. Tags are applied automatically — you can edit or delete them at any time.

💡
If Ollama is slow or unavailable, bookmarks are still saved — they just won't have AI tags until Ollama is reachable. You can trigger re-tagging from the bookmark's context menu.

⚠️ Known Breakpoints
🔴 Meilisearch index becomes corrupt or out of sync
If search stops returning results or returns errors, the Meilisearch index may need rebuilding. This is safe — the source of truth is Hoarder's own database, not Meilisearch. 
cd ~/hoarder
docker compose restart meilisearch
docker compose restart web
# Hoarder will re-index automatically on restart.
If that doesn't fix it, wipe the index volume and let it rebuild: 
docker compose down
rm -rf ./data/hoarder_meili
docker compose up -d
🟡 Domain change breaks OIDC callback
If you change your domain, you must update three places:
  1. Authentik provider redirect URI → → Providers → Hoarder
  2. NEXTAUTH_URL in ~/hoarder/docker-compose.yml
  3. OAUTH_WELLKNOWN_URL in ~/hoarder/docker-compose.yml
Then docker compose up -d --force-recreate web to apply.
🟡 AI tagging stops working after Ollama model changes
If you remove or rename the llama3.1 or llava models from Ollama, Hoarder's tagging will silently fail. Bookmarks will still save but will have no AI tags. Update INFERENCE_TEXT_MODEL and INFERENCE_IMAGE_MODEL in the compose file to match the models currently available in Ollama, then restart the web container.
🟢 Chrome container logs fill up
The headless Chrome container generates verbose debug logs. These don't affect functionality but can bloat Docker log storage over time. Prune periodically: docker compose logs chrome --tail=0 or set a log limit in compose with logging: driver: "json-file" options: max-size: "10m".

🎯 How Hoarder Achievements Work

Hoarder does not have native webhooks. Achievements are driven by polling Hoarder's REST API periodically and checking bookmark counts, tag usage, and list membership. The achievement server makes authenticated API calls using a per-user API key generated in Hoarder settings.

💡
Each user needs to generate an API key in Hoarder (Settings → API Keys → Create) and provide it to the achievement server config so it can poll on their behalf.

💾 Where Data Lives

All Hoarder data is stored inside the WSL2 filesystem under ~/hoarder/data/. There is no per-user folder in for this service — all user data (bookmarks, tags, lists) is stored in Hoarder's own SQLite database.

🛠️ Common Issues
❌ All three containers crash-loop on startup
Usually caused by a malformed docker-compose.yml (bad indentation, missing quotes) or a mismatched MEILI_MASTER_KEY between the web and meilisearch services — both must use the exact same key. 
cd ~/hoarder
docker compose logs web --tail=50
docker compose logs meilisearch --tail=50
Verify the MEILI_MASTER_KEY value is identical in both service blocks, then: 
docker compose up -d --force-recreate
❌ OIDC login fails — "Invalid redirect URI"
The redirect URI registered in Authentik must exactly match what Hoarder sends. Go to → Providers → Hoarder and confirm the redirect URI is: 
/api/auth/callback/custom
Also verify NEXTAUTH_URL in docker-compose.yml matches your current domain exactly (no trailing slash). After any change, restart the web container: docker compose up -d --force-recreate web
🟡 Search returns no results / search is broken
Meilisearch may have lost its index. Restart both services — Hoarder will trigger a re-index automatically: 
cd ~/hoarder
docker compose restart meilisearch
docker compose restart web
If still broken after 5 minutes, wipe the index volume and let it rebuild from scratch: 
docker compose down
rm -rf ./data/hoarder_meili
docker compose up -d
🟡 AI tags are not being applied to new bookmarks
Check that Ollama is reachable from the WSL2 container. The OLLAMA_BASE_URL must use the server's LAN IP, not localhost (which would point inside the container, not to the Windows host). 
# Test from inside the web container:
docker compose exec web wget -qO- http://[server-lan-ip]:/api/tags
Also confirm that llama3.1 is pulled in Ollama: ollama list. If the model name changed, update INFERENCE_TEXT_MODEL in the compose file.
🟡 Chrome container is not reachable — page snapshots fail
The chrome container exposes its remote debugging port on 9222 to the internal Docker network. If it's not reachable, Hoarder will save bookmarks without a snapshot/screenshot. Restart the chrome container: 
cd ~/hoarder
docker compose restart chrome
Verify it's up: docker compose ps chrome should show running.
🟢 Hoarder UI is slow on first load after server restart
Meilisearch takes 10–20 seconds to warm up after a container restart before it can serve search queries. The web UI may feel sluggish during this window. This is normal — wait a moment and reload.
