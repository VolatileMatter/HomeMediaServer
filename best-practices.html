// Adding a New Service · Integration Standards · The Rules //

A reference document — for yourself or any AI assistant — describing exactly how any new service must be configured to fit this server's architecture. Print or share this when adding something new.

🐉 The Philosophy of This Server

This server exists for one reason: I control everything. I preserve everything. The server is a bulwark against predatory corporate practices — companies removing content we paid for, shutting down services, region-locking media, or simply going out of business. Everything here is self-hosted, self-owned, and self-controlled.

My family has indefinite access to the books, movies, music, and games we love — regardless of what any company decides to do with "their" platform. No subscriptions required. No accounts with third parties. No data leaving this building except through channels we control.

Every service added must serve this philosophy. If it requires a cloud account, sends data home, or can be remotely disabled by a corporation — it does not belong here.

📋 New Service Integration Checklist
💡
Use this checklist every time you add a service. Every item must be addressed — either implemented or explicitly decided against with a reason documented.
1 · Philosophy Alignment
Service runs entirely on-premises — no cloud dependency required to function
Data is stored locally and I own it completely (not locked to a proprietary format)
Service cannot be remotely disabled or "bricked" by its vendor
Preferably open-source or at minimum has been audited for telemetry / data exfiltration
If telemetry exists, it is disabled in configuration before the service goes live
2 · Port Assignment
Check Ports tab — confirm the default port does not collide with any existing service
If collision: assign a non-conflicting port and document it in the Ports tab with a ⚠️ MOVED note
Add the new port to ports.html with: port, service name, protocol, public/Tailscale/localhost access level, and which tab documents it
Open Windows Firewall for the port if LAN access needed: netsh advfirewall firewall add rule...
3 · Authentik SSO Integration
Option A (preferred): Native OIDC — if the service supports OIDC/OAuth2, create an Authentik Application + OIDC Provider. Set the redirect URI. Paste Client ID/Secret into the service config.
Option B: Caddy forward_auth — if no native OIDC support, add a Caddyfile block with forward_auth localhost:9000 { uri /outpost.goauthentik.io/auth/caddy }. The service is protected even without knowing about Authentik.
Assign the new Authentik Application to the correct groups (Admins / Family / Friends) based on who should have access
Test login with each group level before declaring done
Confirm the nuke procedure works: deactivating the user in Authentik removes access to this service
4 · Cloudflare Tunnel & Domain
Decide access tier: Public (via Cloudflare Tunnel), LAN+Tailscale, or localhost-only
If public: add entry to cloudflared config.yml — hostname: service.yourdomain.com → service: http://localhost:[PORT]
If public: add Cloudflare Access policy restricting to known Gmail addresses (double-layer with Authentik)
Admin/download services (qBittorrent, *arr stack, control dashboard) must be Tailscale-only — never exposed publicly
If public: add the subdomain to the Caddy subdomain routing table in the Ports tab
5 · Per-User Data Storage
Determine: does this service store per-user data (saves, progress, preferences, files)? If no shared library is involved, per-user storage is almost always needed.
Per-user data goes in D:\Users\[username]\[service-name]\ — not anywhere else
For browser-based services: use the X-authentik-username header to resolve the save path directly. No symlink needed.
For Sunshine-streamed apps: add a pre-launch curl call to the Session Manager (port 9004) and add the app's save folder to SYMLINK_MAP in session_manager.py
Add the new subfolder to USER_SUBDIRS in session_manager.py so ensure_user_dirs() creates it for new users
Verify that D:\Users\[username]\[service-name]\ appears under the user's App Data folder in Nextcloud and counts toward their quota
Update SSO tab → User Data Storage → Per-App Data Locations Reference table with the new service row
6 · Auto-Start on Boot
Native Windows service: configure as Automatic startup — sc config ServiceName start=auto
Python/Node/other process: install via NSSM — nssm install ServiceName [executable]. NSSM sets Automatic by default.
Docker in WSL2: add to D:\Server\Scripts\start_docker_services.bat and confirm it's in Task Scheduler (At Startup, delay 2 min)
Add a row to the Restarting tab table: service name, restart method, boot time, AUTO/MANUAL badge
Add a monitor to Uptime Kuma (port 3001) — HTTP or TCP check, 1-minute interval
7 · User Homepage & Opt-In
Add the service to the App Library in user-homepage.html — new services are off by default. Users enable them individually.
Define: { id, icon, name, desc, url, blk (feed block ID or null), locked:false, def:false, ql }
If the service shows activity (recently played, continue watching, etc.): add a hidden blk-[id] feed block to the homepage that shows when the user enables the service
Add to the All Services tab: create a service card with all required fields (see All Services tab for the template)
8 · Achievements (Optional)
Does this service have natural milestones worth celebrating? (first book read, 10 hours played, etc.) If yes, define achievements.
Add achievements to the achievement server's definitions.json (see Achievements tab)
Add a webhook or API call from the service to POST http://localhost:9003/api/achievement/trigger when the milestone is hit
For games: consider enabling RetroAchievements if the emulator supports it (see Achievements tab)
9 · Documentation
Create a new tab in this binder: [service].html using shared.css
Add the tab to index.html: entry in FILES dict, nav-item button in sidebar, and a pane div
Tab must include: install steps, port, Authentik config method, user data storage path, Cloudflare/access tier, and backup notes
Tab must include a "💾 User Data Storage → SSO TAB" section if any per-user data is involved
Reference the service from the Roadmap (home.html) with a clickable link to the new tab
📜 The Non-Negotiable Rules
I
Everything behind Authentik
No service is accessible without first passing through Authentik authentication. Either native OIDC or Caddy forward_auth. No exceptions. Not even "just temporarily."
II
All user data in D:\Users\[username]\
Per-user data — saves, progress, preferences, files — lives in the user's personal folder and nowhere else. It counts toward their Nextcloud quota. It is backed up as part of their profile. It is deleted when they are nuked.
III
New services are opt-in, not opt-out
When a new service is added, it does not appear on users' homepages automatically. Users must go to the App Library and enable it. Only Jellyfin is on by default.
IV
Ports are documented before services go live
Every port must be recorded in the Ports tab before the service is deployed. Collision checking is mandatory. No service runs on an undocumented port.
V
Admin/download tools are Tailscale-only
qBittorrent, the *arr stack, the control dashboard, and any service that could be used to exfiltrate data or damage the server's operation must not be exposed publicly — not even behind Authentik. Tailscale only.
VI
Services must auto-restart after reboot
Every always-on service must be configured to restart automatically after a Windows reboot. NSSM for processes, Windows Service for native installs, Task Scheduler for WSL2/Docker. The Restarting tab must be updated.
VII
I preserve everything — backups are mandatory
Every service with meaningful data (worlds, libraries, configurations) must have a documented backup strategy. Weekly automated backups to a secondary location minimum. Data that cannot be recovered is data that doesn't exist.
VIII
No cloud lock-in
A service that requires an active cloud account or internet connection to function has no place here. If the vendor's servers go dark tomorrow, the service must continue to work. Check the data format — it must be open or exportable.
🔧 Quick Config Templates
Caddyfile — new service with Authentik forward_auth
service.yourdomain.com {
    forward_auth localhost:9000 {
        uri /outpost.goauthentik.io/auth/caddy
        copy_headers X-authentik-username
                     X-authentik-groups
                     X-authentik-email
        trusted_proxies private_ranges
    }
    reverse_proxy localhost:[PORT]
}
Session Manager — add new symlink entry
# In session_manager.py, add to SYMLINK_MAP:
{
  'link':   'D:/[App]/active_data',
  'subdir': '[service-name]',
},

# Add to USER_SUBDIRS:
'[service-name]',

# Restart session manager:
nssm restart SessionManager
Cloudflare Tunnel — new public entry
# In cloudflared config.yml:
ingress:
  - hostname: service.yourdomain.com
    service: http://localhost:[PORT]
    originRequest:
      noTLSVerify: false
  # ... existing entries
  - service: http_status:404

# Restart tunnel:
cloudflared service restart
Homepage App Library — new entry
// In user-homepage.html APPS array:
{
  id:      'myservice',
  icon:    '🔧',
  name:    'My Service',
  desc:    'What it does in 5 words',
  url:     'http://localhost:[PORT]',
  blk:     'blk-myservice', // or null
  locked:  false,
  def:     false,    // never true (opt-in)

},
