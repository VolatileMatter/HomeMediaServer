// Adding a New Service · Integration Standards · The Rules //

A reference document — for yourself or any AI assistant — describing exactly how any new service must be configured to fit this server's architecture. Print or share this when adding something new.

🐉 The Philosophy of This Server

This server exists for one reason: I control everything. I preserve everything. The server is a bulwark against predatory corporate practices — companies removing content we paid for, shutting down services, region-locking media, or simply going out of business. Everything here is self-hosted, self-owned, and self-controlled.

My family has indefinite access to the books, movies, music, and games we love — regardless of what any company decides to do with "their" platform. No subscriptions required. No accounts with third parties. No data leaving this building except through channels we control.

Every service added must serve this philosophy. If it requires a cloud account, sends data home, or can be remotely disabled by a corporation — it does not belong here.

📋 New Service Integration Checklist
💡
Use this checklist every time you add a service. Every item must be addressed — either implemented or explicitly decided against with a reason documented.
1 · Philosophy Alignment
Service runs entirely on-premises — no cloud dependency required to function
Data is stored locally and I own it completely (not locked to a proprietary format)
Service cannot be remotely disabled or "bricked" by its vendor
Preferably open-source or at minimum has been audited for telemetry / data exfiltration
If any telemetry exists, confirm it can be disabled in config and document how
2 · Port Assignment
Check the Ports tab for conflicts before assigning a port
Record the new port in the Ports tab before the service goes live
Add the port to config.js so all service pages can reference it via data-cfg
Open the firewall rule: netsh advfirewall firewall add rule name="..." dir=in action=allow protocol=TCP localport=[PORT]
3 · Auto-restart
Service restarts automatically after a Windows reboot (NSSM, Windows Service, or Task Scheduler)
Update the Restarting tab in the binder with how this service auto-starts
Test it: reboot the server, confirm the service comes back up without manual intervention
4 · Authentication
Service is behind Authentik — either native OIDC or Caddy forward_auth. No exceptions.
New Authentik application uses the server-users group policy for access control
If public-facing: add Cloudflare Access policy restricting to known Gmail addresses (second layer of auth beyond Authentik)
Admin and management tools (download managers, control dashboards, config UIs) are LAN-accessible only — never exposed publicly, even behind Authentik
If public: add the subdomain to the Caddy routing table and Cloudflare Tunnel config in the Ports tab
5 · Per-User Data Storage
Determine: does this service store per-user data (saves, progress, preferences, files)?
Per-user data goes in D:\Users\[username]\[service-name]\ — not anywhere else
For browser-based services: use the X-authentik-username header to resolve the save path. No symlink needed.
For Sunshine-streamed apps: games are launched through Playnite — the single Sunshine application. The pre-launch curl call to Session Manager (port 9004) is on the Playnite app entry, not individual games. Add each game's save folder to SYMLINK_MAP in session_manager.py as normal.
Add the new subfolder to
Verify D:\Users\[username]\[service-name]\ appears under the user's App Data in Nextcloud and counts toward their quota
Verify that deleting a user (or running user cleanup) removes this folder correctly
6 · Homepage Integration
Service is added to the App Library in user-homepage.html with def: false (opt-in, not default)
App Library entry has a short, accurate description (5 words max) and the correct URL
Quick-link (ql) is defined for the service so it appears in the command palette
If applicable: homepage widget or feed block configured in homepagesetup.html
7 · Achievements & Activity Tracking
Does this service produce events that could trigger achievements? If yes, configure webhooks or polling.
Achievement definitions added to definitions.json for this service
Does this service produce content history useful for the AI recommender? (Media services: yes. Utility services: no.)
If yes: rec_server.py endpoint configured to receive history records from this service
For games: consider enabling RetroAchievements integration if the emulator supports it
Activity tracking respects user opt-out — check that rec_server.py skips opted-out users
8 · Backups
Backup strategy documented in the service's Data Storage tab
Config / database backed up at least weekly to a secondary location
Per-user data included in the user profile backup job
Restore procedure verified: actually test restoring from backup at least once
9 · Documentation
Create a new tab in this binder: [service].html using service-template.html as the base
Add the tab to index.html: entry in FILES dict, nav-item button in sidebar, and a pane div
Tab includes all 7 panes: Setup, Overview, Maintenance, Achievements, Storage, Troubleshooting, UI Preview
All ports, URLs, and paths use data-cfg attributes — no hardcoded values
UI Preview pane has at least a placeholder chrome frame — complete the mock when time allows
Reference the service from the Roadmap (home.html) with a clickable link to the new tab
Service added to services.json catalogue (do not edit services.html directly)
10 · Sunshine-Streamed Game Security
Required for any game added to the Playnite library and streamed via Sunshine.
Sunshine runs as the stream-user Standard Windows account — not your admin account
terminate_on_app_close = true is set in Sunshine Configuration → General (verified in web UI)
Playnite (fullscreen) is configured as the shell replacement for stream-user — not Windows Explorer
AutoHotkey hotkey-blocker script is in stream-user's startup folder (blocks Alt+Tab, Win key, Ctrl+Esc)
Game executable is accessible by stream-user (NTFS ACL grants ReadAndExecute on the game folder)
Game added to Playnite library — NOT as a separate Sunshine application. Moonlight shows one tile only: "Game Library"
E2E test: connect Moonlight → launch game → exit game → verify returns to Playnite, NOT desktop
E2E test: Alt+Tab during a game → verify nothing happens (AHK blocker active)
📜 The Non-Negotiable Rules
I
Everything behind Authentik
No service is accessible without first passing through Authentik authentication. Either native OIDC or Caddy forward_auth. No exceptions. Not even "just temporarily." Not even for services that "only admins use."
II
All user data in D:\Users\[username]\
Per-user data — saves, progress, preferences, files — lives in the user's personal folder and nowhere else. It counts toward their Nextcloud quota. It is backed up as part of their profile. It is deleted when they are removed from the server.
III
New services are opt-in, not opt-out
When a new service is added, it does not appear on users' homepages automatically. Users go to the App Library and enable it themselves. Only Jellyfin is on by default. This keeps homepages from becoming cluttered and gives users agency.
IV
Ports are documented before services go live
Every port must be recorded in the Ports tab and added to config.js before the service is deployed. Collision checking is mandatory. No service runs on an undocumented port. This is how we avoid "what's on port 8080 again?" six months from now.
V
Admin tools are LAN-only
Download managers, the *arr stack, control dashboards, and any service that could be used to exfiltrate data or damage the server's operation must not be exposed publicly — not even behind Authentik. These are accessible from the local network only (server LAN IP). If you need remote access to these tools, use a VPN or SSH tunnel.
VI
Services must auto-restart after reboot
Every always-on service must be configured to restart automatically after a Windows reboot. NSSM for processes, Windows Service for native installs, Task Scheduler for WSL2/Docker. The Restarting tab must be updated. Test by actually rebooting — don't assume it works.
VII
I preserve everything — backups are mandatory
Every service with meaningful data (worlds, libraries, configurations, user saves) must have a documented backup strategy. Weekly automated backups to a secondary location minimum. Data that cannot be recovered is data that doesn't exist.
VIII
No cloud lock-in
A service that requires an active cloud account or internet connection to function has no place here. If the vendor's servers go dark tomorrow, the service must continue to work. Check the data format — it must be open or exportable.
IX
Activity tracking is opt-out, not opt-in
The achievement system and AI recommendation engine track user activity by default — this is what makes them useful. However, users can opt out of activity tracking at any time in User Settings → Privacy. The rec_server.py and achievement_server.py must always check and respect opt-out flags. Never track a user who has opted out.
X
Config is code — no hardcoded values in HTML
All ports, domains, paths, and version strings in service documentation must use data-cfg attributes that resolve from config.js. When the server IP changes, or a port is reassigned, config.js is the single place to update. Hardcoded values in HTML become wrong and stay wrong.
XI
Game streaming runs in a hardened sandbox — users never see the desktop
Sunshine must run as the stream-user Standard Windows account (not admin). Playnite fullscreen is the only Sunshine application — it replaces Explorer.exe as the shell for stream-user. terminate_on_app_close = true must be set so that if Playnite exits, the stream disconnects immediately rather than exposing the Windows desktop. Alt+Tab and system hotkeys are blocked via an AutoHotkey startup script. No exceptions — a user who escapes to the desktop has unrestricted access to the server.
🔧 Quick Config Templates
Caddyfile — new service with Authentik forward_auth
service.yourdomain.com {
    forward_auth localhost:9000 {
        uri /outpost.goauthentik.io/auth/caddy
        copy_headers X-authentik-username
                     X-authentik-groups
                     X-authentik-email
        trusted_proxies private_ranges
    }
    reverse_proxy localhost:[PORT]
}

# In session_manager.py, add to SYMLINK_MAP:
{

  'subdir': '[service-name]',
},

# Add to USER_SUBDIRS:
'[service-name]',

# Restart session manager:
nssm restart SessionManager
Sunshine kiosk — add a game to Playnite (not to Sunshine)
# Do NOT add individual games as Sunshine applications.
# Sunshine has ONE app: "Game Library" (Playnite fullscreen).
# Add games inside Playnite instead:
#   Playnite -> Add Game -> Manual -> set Name, Platform, Executable
#   Or use Playnite's built-in library scanners for Steam/GOG/emulators

# Grant stream-user access to the game folder (PowerShell):
$acl = Get-Acl "D:\Games\MyGame"
$rule = New-Object System.Security.AccessControl.FileSystemAccessRule(
    "stream-user","ReadAndExecute","ContainerInherit,ObjectInherit","None","Allow")
$acl.SetAccessRule($rule)
Set-Acl "D:\Games\MyGame" $acl

# [x] terminate_on_app_close = true in Sunshine web UI
# [x] Playnite is stream-user's shell (HKCU Winlogon\Shell)
# [x] AHK blocker in stream-user startup folder
# [x] Alt+Tab tested and confirmed blocked
