Restarting Services

// Post-Reboot Checklist · What's Automatic · What Needs a Hand //

After a Windows update reboot (or any planned restart), most services come back up on their own within a few minutes. A small number require manual action — mostly game servers. This page is your checklist.

🐉
The goal: All always-on services restart automatically. Game servers (Minecraft, ARK, SS13) are intentionally off by default after a reboot — you turn them on when you want a session. Emulators (Dolphin, Xenia) are on-demand anyway and need no automation. Sunshine starts automatically so remote streaming is always available.

🔧 Setting Up Automatic Restarts
All NSSM services — how to set them to auto-start

Any service installed via NSSM (nssm install ServiceName ...) is already configured as a Windows Service. By default NSSM sets startup type to Automatic — they start on boot. Verify with:

sc query ServiceName      # shows current state
sc qc ServiceName         # shows startup type (should say "AUTO_START")

If any service shows DEMAND_START, fix it:

sc config ServiceName start=auto
nssm set ServiceName Start SERVICE_AUTO_START
WSL2 Docker services — Task Scheduler startup task

Authentik, Immich, RomM, Mealie, and LinguaCafe all run in Docker inside WSL2. They need a Task Scheduler entry to start on boot:

# Create D:\Server\Scripts\start_docker_services.bat:
wsl -d Ubuntu -- bash -c "cd ~/authentik && docker-compose up -d >> ~/startup.log 2>&1"
wsl -d Ubuntu -- bash -c "cd ~/immich && docker-compose up -d >> ~/startup.log 2>&1"
wsl -d Ubuntu -- bash -c "cd ~/romm && docker-compose up -d >> ~/startup.log 2>&1"
wsl -d Ubuntu -- bash -c "cd ~/mealie && docker-compose up -d >> ~/startup.log 2>&1"
wsl -d Ubuntu -- bash -c "cd ~/linguacafe && docker-compose up -d >> ~/startup.log 2>&1"
# In Task Scheduler:
# Name: Start Docker Services
# Trigger: At startup, delay 2 minutes (WSL2 needs Windows to fully boot first)
# Action: Run start_docker_services.bat
# Run whether user is logged on or not: ☑
# Run with highest privileges: ☑
qBittorrent — must start AFTER Mullvad VPN

qBittorrent is bound to the Mullvad network interface. If it starts before Mullvad connects, it won't be able to bind and will either fail to start or fall back to your real IP.

# Create D:\Server\Scripts\start_qbittorrent.bat:
@echo off
:wait_vpn
REM Wait until Mullvad shows "Connected" before starting qBittorrent
mullvad status | findstr /C:"Connected" >nul 2>&1
if errorlevel 1 (
    timeout /t 5 /nobreak >nul
    goto wait_vpn
)
REM VPN is connected — start qBittorrent
nssm start qBittorrent

# In Task Scheduler:
# Name: Start qBittorrent after VPN
# Trigger: At startup, delay 3 minutes (gives Mullvad time to connect)
# Action: Run start_qbittorrent.bat
📋 Post-Reboot Checklist (interactive)

Run through this after any server reboot. Check each item off as you verify it.

  • Wait 5 minutes after boot before checking anything
    Windows services, WSL2, and Mullvad all need time to initialize. Checking too early gives false negatives.
  • Open Uptime Kuma at http://[server]:3001 — verify green across the board
    This is the fastest way to see what's up vs down. Any red lights need investigation.
  • Verify Mullvad VPN is connected
    Open Mullvad app on the server. Must show "Connected" before qBittorrent does anything useful.
  • Check Authentik is accessible at https://auth.yourdomain.com
    WSL2 Docker takes 2–3 minutes to start all Authentik containers. If it's still starting, wait another minute.
  • Verify Jellyfin loads at https://movies.yourdomain.com
    Try browsing to something. If Authentik isn't up yet, Jellyfin login will fail — check Authentik first.
  • Check qBittorrent Web UI at http://[tailscale-ip]:8090
    Should show active torrents resuming automatically. If no connection, check if Mullvad is connected first.
  • Verify Sunshine is running — try connecting with Moonlight from another device
    Sunshine starts automatically. If Moonlight can't connect, check that Sunshine shows up in Windows Services.
  • Start Minecraft servers if a session is planned
    Use the Dashboard tab → Start buttons, or run the Task Scheduler tasks manually.
  • Check the Dashboard → Active Sessions for any anomalies
    Make sure no ghost sessions are shown from before the reboot.
⚠️ What Must Always Be Done By Hand
🎮 Game Servers (Minecraft, ARK, SS13)

Intentionally manual. Game servers shouldn't auto-start because they consume significant RAM. You start them when a gaming session is planned, and they shut down at 2am via the scheduled task.

Exception: if Minecraft is heavily used and you want it always on, you can make it automatic via NSSM. Just be aware it holds ~2–4GB RAM.

🕹️ Dolphin, Xenia, Steam

On-demand emulators and Steam launch only when someone actively wants to play — they're not services. Sunshine launches them when a user connects via Moonlight. No automation needed or desired.

🔑 Windows Login Required for Some Services

Services that use Task Scheduler with "Run only when user is logged on" require you to be logged into Windows on the server. For headless operation, either switch them to "Run whether logged on or not" in Task Scheduler, or keep auto-login enabled on the server (set a password-protected screensaver for physical security).

🔍 Uptime Kuma Monitoring Itself

Uptime Kuma starts automatically, but you need to manually check if any alerts fired during the downtime window. Open it after every reboot and review the incident history.
