💿 Step 1 — Choosing and Flashing the OS
Raspberry Pi OS Lite 64-bit vs Ubuntu Server 24.04 LTS — which to use

Both are Debian-based and run Docker identically. The decision comes down to hardware support and longevity.

Raspberry Pi OS Lite 64-bitUbuntu Server 24.04 LTS
Penta SATA HAT support✅ Best — Waveshare's official documentation targets Pi OS. The dtparam=pciex1 overlay works reliably.⚠️ Works but requires manual kernel overlay config — slightly more fiddly on first setup.
Pi 5 hardware optimisation✅ Best — maintained by the Pi Foundation specifically for Pi hardware. VideoCore GPU, thermal management, and GPIO all work out of the box.Good — Ubuntu follows Pi OS's lead with a short delay for new hardware features.
Docker support✅ Full arm64 support via docker.com's official install script.✅ Full arm64 support, same install script.
Security update cycleRolling updates — well-maintained but no fixed LTS schedule.✅ 5-year LTS (supported until 2029). Clear, predictable support window.
Community resourcesLargest Pi-specific community. Most Penta HAT guides use Pi OS.Larger general Linux community. Better for novel problems.
VerdictUse Raspberry Pi OS Lite 64-bit. For a device running a HAT with PCIe attached drives 24/7, native hardware support matters more than Ubuntu's LTS cycle. If the Pi ever runs headlessly for years without intervention, Pi OS's hardware-specific tuning is an advantage.
Flash Raspberry Pi OS Lite (64-bit) with Raspberry Pi Imager

Download Raspberry Pi Imager. Select: Raspberry Pi 5 → Raspberry Pi OS Lite (64-bit). Before writing, click the ⚙️ gear icon and configure:

  • Hostname: sentry
  • Username: ixora
  • Set a strong password
  • Enable SSH → use password authentication (or paste your public key for key-based auth)
  • Locale: set your timezone (America/Los_Angeles for San Diego)

If booting from NVMe: flash to the NVMe drive using a USB-to-NVMe adapter, then set boot order in raspi-config after first boot from SD, or use the Pi Imager to write directly to NVMe.

First boot — update and install essentials 
# Connect via SSH:
ssh ixora@sentry.local
# Or by IP if mDNS isn't resolving: ssh ixora@192.168.1.20

sudo apt update && sudo apt full-upgrade -y
sudo apt install -y git curl wget vim htop net-tools samba samba-common-bin smartmontools ntfs-3g cifs-utils

Set a static LAN IP via your router

Router admin → DHCP reservations → find the Pi's MAC address → assign 192.168.1.20 permanently. Reboot the Pi to pick up the reservation. Update nodes.json → nodes[pi].lanIP once confirmed. A static IP is essential — the PC's SMB mount and power manager scripts must always reach the Pi at the same address.

🌡️ Step 2 — Cooling (Do This Before Installing the HAT)
Why cooling is non-negotiable on the Pi 5

The Pi 5 runs a significantly more powerful SoC than previous generations. Under sustained Docker load (multiple containers, SMB transfers, Nextcloud activity) it will hit thermal limits without active cooling and throttle the CPU to protect itself. Throttling means stuttering for any active user.

55°C
80°C
85°C
TemperatureWhat happens
Under 55°CNormal operation. Full 2.4GHz CPU speed.
55°C → 80°CFan (if present) ramps up. Performance unaffected.
80°CCPU begins throttling frequency. You'll notice slower Docker builds and sluggish responses.
85°CHard throttle — CPU drops to ~600MHz. Services stall. Avoid at all costs.
Cooling options — ranked
OptionIdle tempLoad tempNotes
Official Raspberry Pi Active Cooler (~$5)~35°C~55°C✅ Best value. Clips directly onto the BCM2712 chip and LPDDR5. 5V PWM fan controlled by the Pi's firmware. Fits under most HATs with standoffs.
Argon ONE V3 M.2 case~38°C~60°CGood all-in-one case with M.2 NVMe slot. Fan is programmable. No room for Penta SATA HAT — incompatible.
Passive heatsink only~45°C~75–80°CInsufficient under sustained Docker load. Throttle risk. Not recommended for 24/7 server use.
No cooling~50°C~85°C+❌ Will throttle under any sustained load. Do not use for server applications.
💡
With the Penta SATA HAT: The HAT sits on top of the Pi via GPIO header and PCIe ribbon. The official active cooler can be installed under the HAT using extended standoffs (usually included with the HAT). Confirm the clearance before ordering. Some Penta HAT kits include their own heatsink for the Pi — check what's in the box.
Install the active cooler before attaching the HAT

Peel the thermal pad backing from both heatsink pads on the cooler. Press firmly onto the BCM2712 (main chip) and LPDDR5 RAM chip. Connect the fan's JST connector to the Pi's fan header (labeled FAN on the board). The firmware controls fan speed automatically — no software configuration needed.

Monitor temperature after setup 
# Check current CPU temperature:
vcgencmd measure_temp

# Watch it live while running workloads:
watch -n 2 vcgencmd measure_temp

# Check CPU frequency (throttling shows as lower frequency):
vcgencmd measure_clock arm
# 2400000000 = 2.4GHz = full speed; lower = throttled

# Install and run a thermal stress test to confirm cooling is adequate:
sudo apt install stress
stress --cpu 4 --timeout 60s
# Temperature should plateau under 70°C with good cooling
💾 Step 3 — Install the Penta SATA HAT and Mount Drives
Attach the HAT

Power off the Pi. Attach the Penta SATA HAT to the 40-pin GPIO header. Connect the PCIe ribbon cable from the HAT to the Pi 5's PCIe FFC connector (white flip-lock connector near the USB ports). Connect your HDDs to the SATA ports. Plug the HAT's 12V power supply into its barrel connector — the HAT and drives need their own power, independent of the Pi's USB-C power supply.

⚠️
The PCIe ribbon cable orientation matters. The gold contacts face a specific direction depending on the HAT model. Check Waveshare's documentation for your exact HAT revision — inserting it backwards does nothing (not destructive, just won't work).
Enable PCIe in the Pi's firmware config 
sudo nano /boot/firmware/config.txt

# Add these two lines at the bottom:
dtparam=pciex1
dtparam=pciex1_gen=3

# Save and reboot:
sudo reboot

# After reboot, confirm the PCIe bus sees the drives:
lsblk
# You should see /dev/sda, /dev/sdb, etc. for each connected drive
Identify drives and check their health before formatting 
# See all drives with sizes:
lsblk -o NAME,SIZE,TYPE,MOUNTPOINT,FSTYPE

# Check SMART health on each drive (never skip this for new or used drives):
sudo smartctl -H /dev/sda
# "SMART overall-health self-assessment test result: PASSED" is what you want

# Full details including temperature and reallocated sector count:
sudo smartctl -a /dev/sda | grep -E "Reallocated|Temperature|Hours"
# Reallocated_Sector_Ct above 0 = the drive has recovered bad sectors — watch it closely
Partition and format new drives

Use ext4 for simplicity and reliability. btrfs offers snapshots and checksumming but is more complex to manage for a family server. ext4 is the right choice here.

# For each new drive (repeat for /dev/sdb, /dev/sdc, etc.):
sudo parted /dev/sda --script mklabel gpt mkpart primary ext4 0% 100%
sudo mkfs.ext4 -L archive /dev/sda1
# The -L flag sets a label so you can reference it by name instead of UUID
Create mount points and configure auto-mount via fstab 
# Single drive setup:
sudo mkdir -p /mnt/archive

# Get UUID (more reliable than device name, which can change):
sudo blkid /dev/sda1
# Note the UUID value, e.g.: UUID="a1b2c3d4-e5f6-..."

# Add to /etc/fstab:
sudo nano /etc/fstab
# Add this line:
UUID=YOUR-UUID-HERE  /mnt/archive  ext4  defaults,noatime,nofail  0  2

# The 'noatime' flag reduces unnecessary writes (extends drive life).
# The 'nofail' flag means the Pi boots normally even if the drive is disconnected.

# Apply without rebooting:
sudo mount -a

# Verify:
df -h /mnt/archive   # should show the drive's full capacity
For multiple drives: set up mergerfs to pool them as one volume

If you have two or more SATA drives and want them to appear as a single /mnt/archive, mergerfs pools them transparently. New files automatically land on the drive with the most free space.

# Install mergerfs:
sudo apt install mergerfs

# Mount individual drives to separate paths first:
sudo mkdir -p /mnt/disk1 /mnt/disk2

# Add each drive to fstab at /mnt/disk1, /mnt/disk2, etc. (same UUID method above)

# Then add a mergerfs pool entry to fstab:
/mnt/disk1:/mnt/disk2  /mnt/archive  fuse.mergerfs  defaults,allow_other,use_ino,category.create=mfs,fsname=mergerfs  0  0

# Reload:
sudo mount -a
df -h /mnt/archive   # should show combined capacity
💡
mergerfs is NOT RAID — it does not provide redundancy. If one drive fails, you lose the files on that drive. For redundancy, pair mergerfs with SnapRAID, which adds parity protection without the overhead of traditional RAID.
Create the archive folder structure 
sudo mkdir -p /mnt/archive/{Movies,TV,Music,Books,Audiobooks,Comics,DnD,Photos,Roms,Backups,Docker}
sudo mkdir -p /mnt/archive/Roms/{gba,gbc,gb,nes,snes,n64,ps1,nds,psp,sega-genesis,mame}

# Set ownership so the ixora user (Docker containers run as uid 1000) can write:
sudo chown -R ixora:ixora /mnt/archive
sudo chmod -R 755 /mnt/archive

📁 Step 6 — SMB Share: Making the Archive Visible to the PC
Configure Samba on the Pi 
sudo nano /etc/samba/smb.conf

# Add at the bottom:
[archive]
   path = /mnt/archive
   browseable = yes
   read only = no
   guest ok = no
   valid users = ixora
   create mask = 0664
   directory mask = 0775

# Set the Samba password for ixora:
sudo smbpasswd -a ixora

# Restart Samba:
sudo systemctl restart smbd nmbd
sudo systemctl enable smbd nmbd
Map Z:\ on the Windows PC 
# In PowerShell (run as Administrator):
# Using IP is more reliable than hostname:
New-PSDrive -Name "Z" -PSProvider FileSystem `
  -Root "\\192.168.1.20\archive" -Persist -Credential (Get-Credential)

# When prompted, enter:  ixora  /  [your Samba password]

# Verify it mapped:
Get-PSDrive Z

Or via File Explorer: This PC → Map network drive → Z: → \\192.168.1.20\archive → ☑ Reconnect at sign-in → ☑ Connect using different credentials.

Add Z:\ as Jellyfin library paths on the PC

Now that the archive is mapped as Z:\, Jellyfin on the Windows PC can use it as a media library. In Jellyfin Dashboard → Libraries → Add Media Library:

Movies library:   Z:\Movies
TV Shows library: Z:\TV

# Or use both PC-local AND Pi paths simultaneously:
Movies: D:\Media\Movies  (PC-local, for fast local access)
        Z:\Movies         (Pi archive, for overflow storage)
# Jellyfin merges both paths into one Movies library.
💡
When Jellyfin streams from Z:\Movies, the video data travels: Pi drive → Pi CPU → Gigabit Ethernet → PC → Jellyfin → Caddy → Tailscale → client. This adds ~1–3ms latency vs local disk and is completely fine for streaming. For 4K HEVC, your bottleneck is the GTX 1070 Ti's transcoding capacity, not the network.

🏛️ Family Digital Vault — Philosophy
🛡️ Sovereignty

No subscriptions. No "delisting" of content. No data mining. Every movie, game, book, and photo is owned outright and stored on hardware you control. Nothing disappears because a streaming deal expired.

⚡ Efficiency

The Pi Sentry (~10W) runs 24/7 and handles everything that doesn't need a GPU. The Windows PC (~200W) only powers on when someone wants Jellyfin or game streaming — cutting idle electricity to near zero overnight.

📂 Organization

Centralized libraries for all ages, accessible from any device on the network. One unified catalog for ROMs, one for movies, one for books — no scattered drives or duplicated folders.

🔄 Longevity

The Penta SATA HAT supports up to 5 drives. As the family's collection grows, slide in another HDD — no reinstall, no migration. Monthly cold-storage backups protect against drive failure.

🚫 Services That Cannot Move to the Pi

The Pi 5 has no GPU. Several services have hard GPU dependencies or other constraints that make them unsuitable for the Pi regardless of available RAM.

ServiceWhy it stays on the PC
JellyfinHardware transcoding (NVENC/QSV) is the whole reason the PC runs it. CPU-only transcoding on the Pi would produce unwatchable stutter at 1080p for multiple simultaneous viewers.
OllamaGPU inference only. CPU-mode Ollama on the Pi 5 runs ~2–4 tokens/second for a 7B model — unusable for real conversation. Stays with the GTX 1070 Ti.
Open WebUIDepends on Ollama being on the same machine (or close to it via LAN). Moves only if Ollama moves.
ImmichThe ML pipeline (face recognition, CLIP embeddings for smart search) requires GPU for practical indexing speed. On Pi CPU, a 10,000-photo library would take ~8 hours to index and peg the Pi at 100% CPU the entire time.
SunshineGame streaming requires direct GPU access for NVENC encoding. Pi 5's VideoCore VII GPU has no NVENC equivalent and cannot encode at the required bitrates.
StreamarrRequires the Mullvad VPN network namespace configured in WSL2. Architecturally belongs near qBittorrent on the PC.
Hoarder/KarakeepThe arm64 image for the embedded Chromium headless browser has a history of instability on Pi. Low risk to try, but not worth the debugging time for an optional service.
🔒 Services That MUST Run on the Pi

These services are required to stay on the Pi because the Pi is the only always-on node. If they ran on the PC, they would be unavailable whenever the PC is off — which is most of the time.

ServiceWhy it must be on the Pi
CaddyAll *.ixo.lol subdomains route through Caddy. If Caddy is off, no service is reachable — even Pi-hosted ones. Must be always-on.
AuthentikEvery service login goes through Authentik. If Authentik is off, nobody can authenticate to anything. Must be always-on.
AdGuard HomeDNS-level ad blocking for Smart TVs and consoles. If it goes down, DNS resolution for opted-in devices fails. Must be always-on.
Achievement ManagerCentral achievement store for all nodes. Pi tracking agent, PC tracking agent, and future nodes all report here. Must be reachable at all times.

⚠️
Add a second Pi when idle RAM consistently exceeds 12 GB. Peak usage (all services active simultaneously) can approach 12 GB, but this scenario is extremely unlikely in practice — nobody is using every single service at the same moment. The real trigger is if your baseline idle (services just running, nobody using them) exceeds 12 GB. That means you're adding new services faster than the Pi can absorb them comfortably.

Good candidates to migrate to a second Pi first: Stirling-PDF (Java, heaviest), Foundry VTT (Node.js, 300 MB idle), Immich base app (500 MB idle), Nextcloud (400 MB idle with DB).

⚠️
The nofail flag is mandatory. Without it, the Pi hangs at boot for 90 seconds waiting for the PC to respond. With nofail, the mount silently fails when the PC is off and the Pi boots normally. RomM's Xbox 360 section will show as empty until the PC comes online and the mount reconnects.
📦 Monthly Cold Storage Backup

Once a month, plug a spare USB 3.0 drive into the Pi. The Penta SATA HAT occupies the PCIe lane — the Pi's USB 3.0 ports remain free for backup drives. Run the backup script manually or schedule it.

#!/bin/bash
# backup_archive.sh — save at /home/ixora/scripts/backup_archive.sh
# Plug USB backup drive into Pi's USB 3.0 port before running.

BACKUP_DRIVE="/mnt/backup_usb"
SOURCE="/mnt/archive"
LOG="/var/log/archive_backup.log"

sudo mkdir -p $BACKUP_DRIVE
sudo mount /dev/sdb1 $BACKUP_DRIVE   # adjust to actual backup device

echo "=== $(date) — Backup started ===" | tee -a $LOG

rsync -avh --progress --delete \
  --exclude "Docker/" \
  --exclude "Backups/" \
  $SOURCE/ $BACKUP_DRIVE/ 2>&1 | tee -a $LOG

echo "=== $(date) — Backup complete ===" | tee -a $LOG
sudo umount $BACKUP_DRIVE
echo "Drive safely unmounted. Remove backup drive." | tee -a $LOG

The --exclude "Docker/" skips container runtime data (rebuildable). Only irreplaceable media and family files are backed up. The --delete flag keeps the backup in sync with the source (removes deleted files from backup too).

🔧 Expanding Storage (Adding More Drives)

The Penta SATA HAT supports up to 5 drives. Adding more is straightforward — shut down the Pi, connect the new drive, boot, and then either use it as independent overflow storage or add it to the mergerfs pool to grow the transparent archive volume.

# For a new drive added as standalone overflow:
# 1. Partition and format: sudo parted /dev/sdb ... && sudo mkfs.ext4 /dev/sdb1
# 2. Mount at /mnt/disk2 via fstab
# 3. Create symlinks for specific folders: ln -s /mnt/disk2/Movies2 /mnt/archive/Movies2

# For adding to an existing mergerfs pool:
# 1. Partition, format, and add to fstab at /mnt/disk2
# 2. Edit the mergerfs fstab entry to include the new disk:
#    /mnt/disk1:/mnt/disk2:/mnt/disk3  /mnt/archive  fuse.mergerfs  ...
# 3. sudo mount -a — the pool expands immediately, no data migration needed

▶️ Power-On Logic
TRIGGER
Family member taps "Wake PC" on dashboard, or a service that requires the PC (Jellyfin, Sunshine) is requested and the PC is detected offline.
1
Pi runs power_manager.py wake → HTTP to smart plug local API → plug switches ON.
2
PC detects AC power → BIOS setting "Restore on AC Power Loss: Power On" triggers automatic boot.
3
Pi polls PC's Tailscale IP every 5 seconds (up to 120s timeout) until it responds.
4
Dashboard PC status dot turns green. User can proceed to Jellyfin/Sunshine.
⏹️ Power-Off Logic
TRIGGER
Midnight cron job, OR no active Jellyfin sessions and no Sunshine sessions for 60 consecutive minutes.
1
Pi queries Jellyfin API: /Sessions?api_key=KEY — counts sessions with active NowPlayingItem.
2
Zero active sessions → Pi SSHs to PC: shutdown /s /t 60 (60-second grace period).
3
Pi waits 90 seconds, confirms PC is offline via Tailscale ping.
4
Pi cuts smart plug power → zero vampire draw from PC.
🐍 Power Manager Script
# Install Tapo local API library:
pip3 install PyP100

# Save script at /home/ixora/scripts/power_manager.py
# chmod +x /home/ixora/scripts/power_manager.py
#!/usr/bin/env python3
"""power_manager.py — Smart plug control for the Windows PC."""

import subprocess, time, sys, requests

PLUG_IP          = "192.168.1.XXX"   # Smart plug LAN IP — update this
PLUG_EMAIL       = "tapo@email.com"  # Tapo account email
PLUG_PASSWORD    = "tapo_password"   # Tapo account password
PC_TAILSCALE_IP  = "100.X.X.X"       # PC Tailscale IP — from nodes.json
PC_LAN_IP        = "192.168.1.10"    # PC LAN IP
PC_SSH_USER      = "ixora"           # Windows username for SSH
JELLYFIN_API_KEY = "YOUR_API_KEY"    # Jellyfin Dashboard → Advanced → API Keys
JELLYFIN_PORT    = 8096
BOOT_TIMEOUT     = 120               # Seconds to wait for PC to boot

def plug(state):
    from PyP100 import PyP110
    p = PyP110.P110(PLUG_IP, PLUG_EMAIL, PLUG_PASSWORD)
    p.handshake(); p.login()
    p.turnOn() if state == "on" else p.turnOff()
    print(f"Plug {'ON' if state == 'on' else 'OFF'}")

def pc_online():
    r = subprocess.run(["ping", "-c", "1", "-W", "2", PC_TAILSCALE_IP], capture_output=True)
    return r.returncode == 0

def active_sessions():
    try:
        r = requests.get(f"http://{PC_LAN_IP}:{JELLYFIN_PORT}/Sessions",
                         params={"api_key": JELLYFIN_API_KEY}, timeout=5)
        return sum(1 for s in r.json() if s.get("NowPlayingItem"))
    except Exception:
        return 0

def wake():
    if pc_online(): print("PC already online."); return True
    print("Waking PC via smart plug...")
    plug("on")
    for i in range(BOOT_TIMEOUT // 5):
        time.sleep(5)
        if pc_online():
            print(f"PC online after {(i+1)*5}s"); return True
    print("WARNING: PC did not come online within timeout.")
    return False

def shutdown():
    if not pc_online(): print("PC already offline."); plug("off"); return
    print("Initiating shutdown...")
    subprocess.run(["ssh", "-o", "StrictHostKeyChecking=no",
                    f"{PC_SSH_USER}@{PC_TAILSCALE_IP}", "shutdown /s /t 60"])
    time.sleep(90)
    if not pc_online(): plug("off"); print("PC shutdown. Plug cut.")
    else: print("WARNING: PC still online after shutdown command.")

def idle_check():
    if not pc_online(): return
    if active_sessions() > 0: print(f"{active_sessions()} active — not shutting down."); return
    print("No active sessions — shutting down.")
    shutdown()

COMMANDS = {"wake": wake, "shutdown": shutdown, "idle": idle_check,
            "on": lambda: plug("on"), "off": lambda: plug("off")}
cmd = sys.argv[1] if len(sys.argv) > 1 else None
if cmd in COMMANDS: COMMANDS[cmd]()
else: print("Usage: power_manager.py [wake|shutdown|idle|on|off]")

🛠️ Common Issues
❌ Pi won't detect SATA drives after boot (lsblk shows no /dev/sda)
Check in order: (1) PCIe ribbon cable seated correctly — gold contacts face the right direction per Waveshare's docs for your HAT revision. (2) dtparam=pciex1 is in /boot/firmware/config.txt. (3) HAT has its own 12V power supply connected and switched on. (4) Run dmesg | grep -i ahci — you should see the SATA controller being detected. If not, the PCIe link isn't up.
❌ Pi hangs at boot for 90+ seconds (drive mount timeout)
An fstab entry is missing nofail. Every drive entry in /etc/fstab must have nofail — especially the CIFS mount for the PC's Xenia folder. Edit fstab and add it. Boot into recovery mode if you can't get a normal shell: append init=/bin/sh to the kernel command line in the Pi's bootloader, or mount the SD card on another machine.
❌ PC won't auto-boot when smart plug turns on
BIOS "Restore on AC Power Loss" must be set to Power On (not Last State, not Power Off). Enter BIOS on next manual boot → Power Management → AC Power Recovery → Always On. This setting path varies by motherboard — search "[your motherboard model] AC power recovery BIOS".
⚠️ CPU is throttling (performance feels slow)
Run vcgencmd measure_temp and vcgencmd measure_clock arm. If temp is above 80°C or clock is below 2.4GHz, cooling is insufficient. Check that: (1) the active cooler fan is spinning (listen for it), (2) the JST fan connector is seated on the Pi's FAN header, (3) there's airflow around the Pi — don't put it in an unventilated enclosure. Reapply thermal paste if the cooler is more than 2 years old.
⚠️ SMB share (Z:\) disconnects after PC wakes from sleep / reboots
Windows mapped drives don't always reconnect automatically after a reboot. Add a Task Scheduler task on the PC: trigger = At logon, action = net use Z: \\192.168.1.20\archive /persistent:yes. Alternatively, use the Pi's Tailscale hostname in the SMB path so it always resolves correctly even if the LAN IP ever changes.
⚠️ Docker container keeps restarting
docker compose logs --tail=50 CONTAINER_NAME. Most common causes: volume path doesn't exist → mkdir -p /mnt/archive/Docker/SERVICENAME; wrong permissions → sudo chown -R 1000:1000 /mnt/archive/Docker/SERVICENAME; port conflict → sudo ss -tlnp | grep PORT.
💡 How to cancel a PC shutdown in progress
The power_manager.py gives a 60-second grace window. On the PC, open Command Prompt and run shutdown /a to abort. Or from the Pi: ssh ixora@PC_TAILSCALE_IP "shutdown /a".
💡 RomM shows no Xbox 360 games
The PC is probably off and /mnt/pc-xenia isn't mounted. This is expected behaviour. Wake the PC, then: sudo mount /mnt/pc-xenia. RomM rescans on a schedule — or trigger a manual scan in RomM's web UI under Libraries.
