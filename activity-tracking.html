🛡️
Users can opt out of activity tracking at any time. Go to settings.yourdomain.com → Privacy → Activity Tracking and toggle it off. When opted out: no new activity events are collected, the recommendation engine stops updating the user's profile, and achievement triggers are suppressed. Previously collected data can be deleted on request. The achievement_server.py and rec_server.py both check and respect the opt-out flag on every event — an opted-out user is silently skipped, with no error and no data written.
Part 1 — Achievement System

🐍 achievement_server.py
Core server implementation
# \achievement_server.py
# Flask server — port 
from flask import Flask, request, jsonify
import json, os, time

app = Flask(__name__)
DEFS_FILE    = '/definitions.json'
UNLOCKS_FILE = '/unlocks.json'
USERS_BASE   = ''

def load(f): return json.load(open(f)) if os.path.exists(f) else {}
def save(f, d): json.dump(d, open(f,'w'), indent=2)

def is_opted_out(username):
    """Check the user's opt-out flag before processing any event."""
    prefs_path = os.path.join(USERS_BASE, username, 'preferences.json')
    if os.path.exists(prefs_path):
        prefs = json.load(open(prefs_path))
        if prefs.get('tracking_optout', False):
            return True
    return False

@app.route('/api/achievement/trigger', methods=['POST'])
def trigger():
    data     = request.json
    username = data.get('username')
    trigger  = data.get('trigger')
    meta     = data.get('metadata', {})
    if not username or not trigger:
        return jsonify({'error': 'missing fields'}), 400

    # Respect opt-out — silently ignore, no error
    if is_opted_out(username):
        return jsonify({'status': 'skipped', 'reason': 'opted_out'}), 200

    defs    = load(DEFS_FILE)
    unlocks = load(UNLOCKS_FILE)
    if username not in unlocks: unlocks[username] = {}
    newly_unlocked = []

    for ach_id, ach in defs.items():
        if trigger in ach.get('triggers', []) and ach_id not in unlocks[username]:
            req = ach.get('requires', {})
            ok  = all(meta.get(k, 0) >= v for k, v in req.items())
            if ok:
                unlocks[username][ach_id] = {
                    'unlocked_at': time.time(),
                    'metadata':    meta
                }
                newly_unlocked.append(ach_id)

    if newly_unlocked:
        save(UNLOCKS_FILE, unlocks)

    return jsonify({'unlocked': newly_unlocked}), 200

@app.route('/api/feed', methods=['GET'])
def feed():
    """Returns recent community unlocks for the homepage feed."""
    unlocks = load(UNLOCKS_FILE)
    defs    = load(DEFS_FILE)
    events  = []
    for user, achievements in unlocks.items():
        for ach_id, info in achievements.items():
            if ach_id in defs:
                events.append({
                    'user':        user,
                    'ach_id':      ach_id,
                    'name':        defs[ach_id].get('name'),
                    'icon':        defs[ach_id].get('icon'),
                    'unlocked_at': info['unlocked_at']
                })
    events.sort(key=lambda e: e['unlocked_at'], reverse=True)
    return jsonify(events[:20])

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=)

# ── Trigger method reference ──
# webhook              — service fires HTTP POST directly to achievement server
# webhook + counter    — webhook fires on each event; server counts to threshold
# webhook + timestamp  — webhook fires; server checks time metadata (e.g. midnight)
# db poll              — achievement_server polls service DB (Task Scheduler, hourly)
# manual               — admin grants via curl or admin UI

Part 2 — AI Recommendation Engine
🤖 What the Recommender Does

The recommendation engine watches what users read, watch, and play — then uses Ollama (running locally) to suggest what they might enjoy next. It reads two catalogue files (books.json and games.json) and a history file per user, and feeds them into a system-prompted Ollama session. No data leaves the server. No third-party API calls.

The recommendations page (ai.yourdomain.com/recommend or the homepage widget) lets users chat with the engine in natural language: "I want something like Dune but shorter" or "a retro game I can play in 30 minutes."

🔒
Opt-out respected here too. rec_server.py checks the tracking opt-out flag before writing any history record. Opted-out users still get the recommendation chat interface — it just uses an empty history (no personalization). They can still provide explicit preferences in-chat.

🐍 rec_server.py — History Recording Endpoint
# POST /api/history/record
# Called by services when a user finishes (or significantly interacts with) content.
# Payload:
{
  "user":     "username",          ← Authentik username
  "type":     "book|game|movie|audiobook",
  "id":       "item_id",           ← catalogue ID from books.json or games.json
  "title":    "Item Title",
  "signal":   "finished|abandoned|started",  ← see signal quality table below
  "metadata": {
    "percent_complete": 95,        ← for books/audiobooks
    "play_time_minutes": 120       ← for games
  }
}

# rec_server.py checks opt-out before writing:
def record_history(data):
    username = data.get('user')
    if is_opted_out(username):
        return {'status': 'skipped', 'reason': 'opted_out'}

    history_path = os.path.join(USERS_BASE, username, 'rec_history.json')
    history = load(history_path) if os.path.exists(history_path) else []
    history.append({
        'type':      data['type'],
        'id':        data['id'],
        'title':     data['title'],
        'signal':    data.get('signal', 'finished'),
        'metadata':  data.get('metadata', {}),
        'recorded_at': time.time()
    })
    save(history_path, history)
    return {'status': 'ok'}
📶 Signal Quality — What Makes a Good Recommendation Input

Not all events carry equal weight. The rec engine uses the signal field to understand the quality of the interaction. Only strong signals meaningfully influence recommendations.

SignalWhen to sendWeight
finishedBook 90%+, movie/episode watched to end, audiobook 95%+, game played 2+ hoursStrong ✅
startedOpened the item, played <15 min, read <10%Weak — skip unless nothing else available
abandonedStarted but stopped at <30% / <30 minNegative signal — avoid similar content
⚠️
Don't over-fire the history endpoint. Sending "started" events for every item the user briefly previews will dilute the history and produce worse recommendations. Only record meaningful interactions.

💡
Keep descriptions accurate and specific. The Ollama model uses these descriptions directly when deciding recommendations. "A fantasy epic" is useless. "A slow-burn political fantasy with complex world-building and minimal action" gives the model real signal.
🧠 Ollama System Prompt
The system prompt sent to Ollama on each recommendation session
You are a personal librarian and game curator for a private home media server.
You have access to two catalogues: books and games. You also have the user's
history of completed items. Use this history to understand their taste.

Rules:
- Only recommend items from the provided catalogues — never invent titles.
- If the user has already finished an item, do not recommend it again.
- If an item is in their "abandoned" history, avoid similar content unless asked.
- Be specific about WHY you're recommending something — connect it to their history.
- Ask clarifying questions if the request is vague.
- Keep recommendations to 3-5 items per response unless asked for more.

Part 3 — Privacy & Administration
🛡️ Opt-Out System — Implementation Details
How opt-out is stored and checked
# Opt-out preference is stored in:
# \[username]\preferences.json
{
  "tracking_optout": true,       ← set to true when user opts out
  "tracking_optout_date": "...",  ← ISO timestamp of when they opted out
  "feed_private": false,          ← hides achievements from community feed
  "achievements_private": []      ← list of specific achievement IDs to hide
}

# BOTH achievement_server.py AND rec_server.py call is_opted_out() before writing:
def is_opted_out(username):
    prefs_path = os.path.join(USERS_BASE, username, 'preferences.json')
    if os.path.exists(prefs_path):
        return json.load(open(prefs_path)).get('tracking_optout', False)
    return False

# When opted out:
# - No achievement triggers are processed (existing unlocks preserved)
# - No new history records are written to rec_history.json
# - The recommendations chat still works but uses empty history
# - The community feed still shows existing unlocks (unless feed_private is also set)
Data deletion on request
# To delete a user's tracking history (on their request):
# 1. Delete their rec history:
Remove-Item "\[username]\rec_history.json"

# 2. Optionally clear their achievement unlocks from unlocks.json:
# (edit \unlocks.json and remove their key)

# 3. Optionally reset their per-user achievement file:
Remove-Item "\[username]\achievements.json"

# Note: do NOT delete preferences.json — that's where opt-out lives.
