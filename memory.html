Managing Memory

// RAM Budget · Spinning Up & Down · Avoiding OOM //

A Windows gaming desktop typically has 16–32 GB RAM. Most server services are lightweight — the problems come when you stack emulators, ARK, Minecraft modded, and Ollama all at once. This page teaches you to be intentional about what's running.

🐉
The key insight: Your server has two distinct modes — "server mode" (serving media, running Minecraft, downloads) and "gaming mode" (Dolphin, Xenia, ARK). These are largely incompatible at the same time on 16 GB RAM. On 32 GB you have more flexibility. Know your RAM budget.
📊 RAM Budget at a Glance

Approximate RAM usage per service at idle/light use. Stack these to see how close you are to your limit.

🔴 Heavy Hitters — 1–10+ GB each
ARK: Survival Evolved — 6–10 GB
Ollama (large model, e.g. 13B) — 10–20 GB VRAM/RAM
Xenia (Xbox 360) — 2–4 GB
Dolphin (GameCube/Wii) — 1–3 GB
Minecraft Modded (Forge/Fabric) — 3–6 GB
Minecraft Vanilla — 1–2 GB
🟡 Medium — 200 MB–1 GB each
Authentik (Docker) — 600 MB–1 GB
Ollama (small model, 7B) — 5–8 GB VRAM (GPU usually, not RAM)
Immich (Docker) — 500 MB–1 GB
Jellyfin (4K transcoding) — 500 MB–2 GB
Nextcloud/XAMPP — 300–600 MB
🟢 Lightweight — Under 200 MB each

Caddy, Jellyfin (idle/direct-play), Navidrome, Calibre-Web, Audiobookshelf, Kavita, Vaultwarden, Uptime Kuma, qBittorrent (idle), Sonarr/Radarr, Prowlarr, Readarr, Vikunja, Kiwix, Recommendations API, Stats agent — all under 200 MB each, most under 100 MB.

⚫ Windows overhead (unavoidable)

Windows 10 itself uses ~2–3 GB RAM. WSL2 takes ~1 GB reserved for the Linux VM. Total overhead: ~3–4 GB before any services run.

⚠️
16 GB scenario: Windows (3 GB) + always-on services (~3 GB) + Authentik (~1 GB) = ~7 GB baseline. You have ~9 GB free. ARK alone needs 6–8 GB. You cannot run ARK + Minecraft Modded + Dolphin simultaneously on 16 GB. 32 GB is strongly recommended if you plan to run game servers and emulators regularly.
🔄 The Golden Rule: Stop Before You Start

Before launching any heavy process, stop something else that's not actively needed. Build a habit:

🦕Before starting ARK: Stop Minecraft modded (if running) + stop Ollama service + close any emulators
🟣Before launching Dolphin/Xenia: Stop ARK + Minecraft modded (vanilla is fine) + stop Ollama if using large model
🤖Before running a large Ollama model (13B+): Stop ARK, stop Minecraft modded, close emulators
⛏️Before starting Minecraft Modded: Stop ARK, close all emulators, close Ollama if using large model
✅For Minecraft Vanilla + Jellyfin + downloads: This runs fine on 16 GB with no special preparation

Check current RAM usage in PowerShell:
Get-Process | Sort-Object WorkingSet -Descending | Select-Object -First 15 Name, @{N='RAM(MB)';E={[math]::Round($_.WorkingSet/1MB,0)}} | Format-Table -AutoSize
# Or a quick summary:
$os = Get-CimInstance Win32_OperatingSystem
$used = [math]::Round(($os.TotalVisibleMemorySize - $os.FreePhysicalMemory) / 1MB, 1)
$total = [math]::Round($os.TotalVisibleMemorySize / 1MB, 1)
Write-Host "RAM: $used GB used / $total GB total ($([math]::Round($used/$total*100))% used)"
🤖 Ollama Memory Management
The model stays in RAM/VRAM until you explicitly unload it

Ollama loads the model into memory on first use and keeps it there for fast responses. On 16 GB with limited VRAM, a 13B model can eat 10–15 GB of unified memory. Control this explicitly:

# Check what models are currently loaded and how much VRAM/RAM they use:
ollama ps

# Unload a model from memory (frees RAM immediately):
ollama stop llama3.1

# Load a smaller model instead of the large one:
ollama run llama3.2:1b   # 1B param model — only ~1.5 GB
ollama run llama3.1:8b   # 8B param model — ~5 GB (good balance)

# Set a timeout so Ollama auto-unloads after inactivity:
# In Authentik admin → add env var to Ollama service:
OLLAMA_KEEP_ALIVE=5m     # unload after 5 minutes of inactivity
💡
Best practice: Use the 8B model for daily use (good quality, manageable memory). Only load the 13B+ model when you actually need it for complex reasoning — and stop it afterward. Set OLLAMA_KEEP_ALIVE=5m as a safety net.
🎮 Emulator Session Management
Dolphins and Xenia are launched by Sunshine — they close when the session ends

When configured as Sunshine apps, Dolphin and Xenia are launched when a user connects via Moonlight and (should be) terminated when they disconnect. Verify this behavior in Sunshine settings:

# In Sunshine Admin (https://localhost:47990):
# Applications → edit each emulator → set "Command Termination" properly
# The process must fully quit when Moonlight disconnects, not minimize to tray.

# Dolphin: launch with --no-gui flag to avoid tray issues:
dolphin.exe --no-gui --batch -e "D:\Emulation\GameCube\game.iso"

# After session ends, verify no dolphin.exe process is still running:
Get-Process dolphin -ErrorAction SilentlyContinue
Preventing "Orphan" emulator processes that eat RAM forever

Sometimes an emulator crashes or is left running without an active Sunshine session. Add a scheduled task that checks for orphaned emulator processes:

# check_orphan_emus.ps1 — run every 30 minutes via Task Scheduler
$emus = @('dolphin', 'xenia', 'xenia_canary')

foreach ($emu in $emus) {
    $proc = Get-Process $emu -ErrorAction SilentlyContinue
    if ($proc) {
        # Check if Sunshine has an active session
        $sunshine_sessions = Invoke-WebRequest "https://localhost:47990/api/sessions" `
            -UseBasicParsing -SkipCertificateCheck 2>$null
        $has_session = $sunshine_sessions.Content -match '"id"'

        if (-not $has_session) {
            Write-Host "No active Sunshine session — killing orphaned $emu"
            Stop-Process -Name $emu -Force
        }
    }
}
# Task Scheduler:
# Name: Kill Orphan Emulators
# Trigger: Repeat every 30 minutes indefinitely
# Action: powershell -ExecutionPolicy Bypass -File "D:\Server\Scripts\check_orphan_emus.ps1"
🚨 When You're Running Out of RAM
Warning signs to watch for
  • Windows starts using the page file (disk) for RAM — everything slows down dramatically
  • Task Manager shows RAM at 90%+ in the Performance tab
  • Services become unresponsive or crash with out-of-memory errors
  • Uptime Kuma starts showing services flipping down/up rapidly
  • The Minecraft server console shows "java.lang.OutOfMemoryError"
Immediate triage — kill the biggest offenders
# One-liner to free the most memory fast:
# Kill ARK (biggest offender), Minecraft modded, and Ollama:
$killers = @('ShooterGame', 'java', 'ollama_runners')
foreach ($k in $killers) {
    $p = Get-Process $k -ErrorAction SilentlyContinue
    if ($p) { Stop-Process -Name $k -Force; Write-Host "Killed $k" }
}

# Then check free RAM:
$os = Get-CimInstance Win32_OperatingSystem
Write-Host "Free RAM: $([math]::Round($os.FreePhysicalMemory / 1MB, 1)) GB"
Windows page file — set it correctly

The page file is your safety net. If real RAM fills up, Windows spills overflow to disk. It's slow (especially on HDD) but prevents crashes. Set it properly:

# In Windows: System → Advanced System Settings → Performance → Settings
# → Advanced → Virtual Memory → Change
# Uncheck "Automatically manage"
# Set CUSTOM SIZE on your SSD (not your HDD):
#   Initial size: [your RAM in MB]     e.g. 16384 for 16 GB
#   Maximum size: [2× your RAM in MB]  e.g. 32768 for 16 GB
# Apply → OK → Reboot
💡
Put the page file on your NVMe SSD, not your media HDD. An NVMe SSD page file is ~10× faster than an HDD page file, which means the "disk spill" situation is much less painful. 32 GB on the SSD is fine.
Long-term solution: upgrade to 32 GB RAM

DDR4 32 GB kits are inexpensive. This is the single most impactful upgrade for a server that runs game servers alongside emulators:

RAMWhat you can run simultaneously
16 GBServer services + Minecraft Vanilla OR Dolphin — not both heavy ones
32 GBServer services + Minecraft Modded + Dolphin + Ollama 8B simultaneously
64 GBEverything at once including ARK + Ollama 13B + all emulators
📺 The Dashboard Memory Widget

The Dashboard tab shows a live RAM usage bar that polls the stats agent every 30 seconds. When RAM is above 80%, a warning banner appears telling users what to avoid. See the Dashboard Setup tab for the stats agent configuration, and the Overview tab for the full load-warning table.

To add custom memory warnings in the stats agent output:

# In stats_server.py — enrich the /stats response:
@app.route('/stats')
def stats():
    vm = psutil.virtual_memory()
    warnings = []
    if vm.percent > 90:
        warnings.append('🔴 RAM critical — avoid game servers and emulators')
    elif vm.percent > 80:
        warnings.append('🟡 RAM elevated — avoid starting ARK or Minecraft Modded')
    elif vm.percent > 70:
        warnings.append('🟡 RAM moderate — Ollama large models not recommended')

    return jsonify({
        'cpu':       psutil.cpu_percent(interval=1),
        'ram':       vm.percent,
        'ram_free_gb': round(vm.available / 1e9, 1),
        'disk_used': psutil.disk_usage('D:\\').percent,
        'warnings':  warnings,
        'uptime':    int(time.time() - psutil.boot_time())
    })
