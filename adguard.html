AdGuard Home

// DNS-Level Ad Blocker · Port 3000 (UI) / 53 (DNS) · Pi Sentry · Docker · Opt-In Per Device //

AdGuard Home runs as a DNS server on the Pi Sentry, blocking ads and trackers at the network level before they ever reach a device. It is not the default DNS for the whole house — only specific devices that can't run browser extensions (Smart TVs, game consoles, streaming sticks) are pointed at it. PCs and phones use normal DNS and manage their own ad blocking via uBlock Origin.

Pi Sentry Compatibility RUNNING ON PI

AdGuard Home runs on the Pi Sentry as a Docker container.

The Pi's 24/7 uptime is essential here — DNS must always be available. The Pi's static LAN IP (192.168.1.20) is what gets set as the DNS server in router or device settings.

🛡️ V1 Goal: AdGuard Home running on the Pi at port 3000 (UI) with DNS on port 53. Smart TVs, Roku, Fire TV, and game consoles configured to use 192.168.1.20 as their DNS. PCs and phones left on normal DNS — no blanket filtering.
🐳 Step 1 — Deploy AdGuard Home on the Pi

AdGuard Home needs access to port 53 (DNS). On most Linux systems port 53 is held by systemd-resolved — disable it first.

SSH into the Pi and free port 53 
ssh ixora@192.168.1.20

# Disable systemd-resolved's stub listener:
sudo sed -i 's/#DNSStubListener=yes/DNSStubListener=no/' /etc/systemd/resolved.conf
sudo systemctl restart systemd-resolved

# Verify port 53 is free:
sudo ss -tulpn | grep :53
# Should return nothing
Add AdGuard Home to docker-compose.yml

Add this service block to /home/ixora/docker/docker-compose.yml:

  adguard:
    image: adguard/adguardhome:latest
    container_name: adguard
    restart: unless-stopped
    network_mode: host   # Required for DNS — must use host networking
    volumes:
      - /mnt/archive/Docker/adguard/work:/opt/adguardhome/work
      - /mnt/archive/Docker/adguard/conf:/opt/adguardhome/conf
⚠️
network_mode: host is required so AdGuard can bind to port 53 directly. This means it bypasses Docker's port mapping — there is no ports: section needed.
Create data directories and start 
sudo mkdir -p /mnt/archive/Docker/adguard/{work,conf}
sudo chown -R ixora:ixora /mnt/archive/Docker/adguard

cd ~/docker
docker compose up -d adguard
docker compose logs -f adguard   # wait for "AdGuard Home is ready"
Complete the web setup wizard

Open http://192.168.1.20:3000 from your browser. The first-run wizard asks you to:

  • Set admin username and password
  • Confirm DNS listens on 0.0.0.0:53 and web UI on 0.0.0.0:3000

After the wizard, the UI moves to port 80 by default — change it back to 3000 in Settings → General if needed to avoid conflict with other Pi services.

Configure upstream DNS

Settings → DNS Settings → Upstream DNS servers. Use fast, privacy-respecting upstream resolvers:

# Cloudflare (fast, privacy policy):
1.1.1.1
1.0.0.1

# Or Quad9 (blocks malware domains at upstream level):
9.9.9.9
149.112.112.112

Enable Parallel requests for speed. Enable DNSSEC if your upstream supports it.

Add safe search and parental controls (optional)

Settings → DNS Settings → Blocking mode. You can enable Google/YouTube Safe Search, Bing Safe Search, and DuckDuckGo Safe Search here — useful for devices used by children. Leave off for adult devices.

⚠️ YouTube Ads on Smart TVs

DNS-level blocking gets most ads on Smart TVs but YouTube ads are partially resistant. Google serves YouTube ads from the same CDN domains as regular video content, so blocking those domains breaks video playback entirely. AdGuard will block some pre-roll ads (especially third-party ones) but cannot reliably block YouTube's first-party ad injection.

The best YouTube experience on your household is through the Invidious instance, which has no ads by design. For Smart TVs specifically, SponsorBlock is not available at the DNS level — it requires a client that supports it (browser extension, or a client app built with SponsorBlock integration).

🩹 Adding a Whitelist Entry

If a device stops working after pointing it at AdGuard DNS, a legitimate domain is probably being blocked. Check the Query Log, find the blocked domain (shown in red), and add it to the allowlist:

# In the AdGuard UI:
Filters → DNS Allowlist → Add Rule
# Enter the domain:
||example.com^

# Or add via CLI if the UI is inaccessible:
# Edit /mnt/archive/Docker/adguard/conf/AdGuardHome.yaml
# Under: user_rules:
#   - '||example.com^'
# Then restart: docker compose restart adguard

Storage footprint is small — config is a few KB, blocklist cache is 10–50 MB depending on lists enabled, query log is auto-pruned based on your retention setting (default 90 days).

💾 Backup

The only file worth backing up is AdGuardHome.yaml — it contains your entire configuration including blocklists, upstream DNS, auth credentials, and custom rules. The query log and statistics are not worth backing up.

# Back up config:
cp /mnt/archive/Docker/adguard/conf/AdGuardHome.yaml \
   /mnt/archive/Backups/adguard-config-$(date +%Y%m%d).yaml
🛠️ Common Issues
Device can't reach the internet after setting DNS to 192.168.1.20
Check that AdGuard is actually running: docker compose ps adguard. Check port 53 is bound: sudo ss -tulpn | grep :53. If systemd-resolved reclaimed port 53 after a reboot, re-run the DNSStubListener fix from setup.
Port 53 conflict — AdGuard fails to start
sudo systemctl stop systemd-resolved && sudo systemctl disable systemd-resolved. Then set a static DNS for the Pi itself in /etc/resolv.conf: add nameserver 1.1.1.1. Restart AdGuard.
Smart TV app broken after enabling AdGuard DNS
A legitimate domain is blocked. Open AdGuard Query Log, filter by the device's IP, look for red (blocked) entries around the time the app broke. Add the domain to the allowlist.
AdGuard UI is inaccessible at port 3000
AdGuard may have moved its UI to port 80 during setup. Try http://192.168.1.20:80. To force it back to 3000, edit /mnt/archive/Docker/adguard/conf/AdGuardHome.yaml and set bind_port: 3000 under the http: section, then restart.
