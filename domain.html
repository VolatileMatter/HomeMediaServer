Network & Domain

// ixo.lol · Tailscale · Caddy · DDNS · Jellyfin TV Carve-out //

How the server is accessible from outside your home. Tailscale handles 99% of access with zero open ports. One deliberate exception exists for Smart TV Jellyfin apps that cannot run Tailscale.

🐉 Architecture summary: All services run behind Tailscale — WireGuard-encrypted, zero open ports, free with no bandwidth cap. The domain gives services pretty URLs. One port (8096) is forwarded publicly at for Smart TV Jellyfin apps only. Cloudflare is not used.

☁️ Step 1 — Set Up Tailscale
🔐 Create a Tailscale account

Go to tailscale.com → Sign up. Use your Google account (same one as Authentik's admin). During setup you'll choose your network/organization name — use . Your machines will get addresses like server..ts.net.

🔐 Install Tailscale on the server (Windows) 
winget install Tailscale.Tailscale
# Or download from tailscale.com/download/windows

After install, open Tailscale from the system tray → Log In → authenticate with Google. The server joins your Tailnet automatically.

Set it to start with Windows (it does this by default). Confirm it's running: look for the Tailscale icon in the system tray — it should show a green checkmark.

🔐 Enable Tailscale HTTPS (MagicDNS)

In the Tailscale admin panel (tailscale.com/admin/dns):

  • Enable MagicDNS — gives your machines stable hostnames on the Tailnet
  • Enable HTTPS Certificates — Tailscale provisions Let's Encrypt certs for *.ts.net hostnames automatically

After enabling, your server is reachable at server..ts.net from any device on the Tailnet. Caddy will use this for TLS.

🔐 Set ACL rules (hub-and-spoke)

In the Tailscale admin panel → Access Controls, paste the ACL policy from nodes.json → tailscaleACLNotes.policy. This enforces that user devices can reach the hub, the hub can reach friend nodes, but user devices cannot reach friend nodes directly.

⚠️
Until ACL rules are set, the default Tailscale policy allows all devices to reach all other devices. Set ACLs before adding friend nodes to the Tailnet.
Fill in nodes.json once the server is on the Tailnet

Open and update the hub node entry:

# In nodes.json → nodes[0] (hub):
"tailscaleIP":       "100.x.x.x",          ← from Tailscale admin panel → Machines
"tailscaleHostname": "server.ixo.ts.net",   ← confirm in admin panel
"lanIP":             "192.168.x.x"          ← set DHCP reservation in router first
🌍 Step 2 — Configure DNS at Porkbun

All subdomains of point at your server's Tailscale IP. Because these resolve to a Tailscale IP (100.x.x.x), only devices on the Tailnet can actually connect — but the domain gives you pretty URLs and valid TLS certs.

Log in to porkbun.com → your domain → DNS. Create the following A records:

Name (subdomain)TypeValueTTL
@ (root)A[hub Tailscale IP]300
* (wildcard)A[hub Tailscale IP]300
tvA[home public IP — see DDNS below]60
💡
The wildcard record *.ixo.lol covers every subdomain automatically — jellyfin.ixo.lol, music.ixo.lol, all of them — without adding individual records for each service. The only exception is tv.ixo.lol which points at your real home IP instead.
⚙️ Step 3 — Configure Caddy

Caddy listens on port 443 and routes subdomains to local services. It gets TLS certificates automatically via Let's Encrypt using your domain. All services are reachable at service.ixo.lol from any Tailscale device.

Install Caddy: download the binary from caddyserver.com → place at → run as a Windows service via NSSM.

nssm install Caddy "C:\caddy\caddy.exe"
nssm set Caddy AppParameters "run --config C:\caddy\Caddyfile"
nssm set Caddy Start SERVICE_AUTO_START
nssm start Caddy
Caddyfile structure — 
# ── Global options ──────────────────────────────────────────────────
{
    email admin@ixo.lol
}

# ── Authentik SSO (protected by itself — no forward_auth wrapper) ──
auth.ixo.lol {
    reverse_proxy localhost:9000
}

# ── Standard protected service (forward_auth pattern) ───────────────
jellyfin.ixo.lol {
    forward_auth localhost:9000 {
        uri /outpost.goauthentik.io/auth/caddy
        copy_headers X-authentik-username X-authentik-groups X-authentik-email
        trusted_proxies private_ranges
    }
    reverse_proxy localhost:8096
}

# ── TV endpoint — NO forward_auth, Jellyfin native auth only ────────
# This block handles the port-forwarded public exception for Smart TVs.
# Only add this block after configuring the router port forward (Step 5).
tv.ixo.lol {
    reverse_proxy localhost:8096
    # No forward_auth — TV apps authenticate directly to Jellyfin.
    # Jellyfin's own login handles auth for this endpoint.
    # Rate limiting: add Caddy rate_limit plugin if brute-force is a concern.
}

# ── Repeat the protected pattern for every other service ────────────
# music.ixo.lol    → localhost:4533  (Navidrome)
# books.ixo.lol    → localhost:8083  (Calibre-Web)
# audio.ixo.lol    → localhost:13378 (Audiobookshelf)
# photos.ixo.lol   → localhost:2283  (Immich)
# ... etc — see ports.html for the full list
📺 Step 4 — Jellyfin Smart TV Exception
Why this exists

Smart TV Jellyfin apps (LG WebOS, Samsung Tizen, Roku, Fire TV, older Apple TV) cannot install Tailscale. They need a plain HTTP/HTTPS endpoint they can reach directly from the internet. Port 8096 is the one deliberate hole in the otherwise fully Tailscale-gated setup.

⚠️
SSO does not work for TV apps. TV apps use Jellyfin's native username/password login. Each user who needs TV access needs a Jellyfin-native password set by you in the admin panel. See the Jellyfin tab → Setup for instructions.
🔐 Set a static LAN IP for your server

Log into your router → DHCP reservations → find your server's MAC address → assign a permanent IP (e.g. 192.168.1.10). Without this, the port forward breaks when your server reboots and gets a different IP.

Update nodes.json → nodes[0].lanIP once set.

🔐 Create the router port forward

Router admin panel → Port Forwarding → Add rule:

External port:  8096
Internal IP:    [your server's static LAN IP]
Internal port:  8096
Protocol:       TCP

Test it: from a phone on mobile data (not WiFi), go to http://[your-home-ip]:8096. You should see the Jellyfin login page.

🔐 Open Windows Firewall for port 8096

The firewall rule for Jellyfin already exists from initial setup. Confirm it's allowing inbound TCP on 8096:

netsh advfirewall firewall show rule name="Jellyfin" dir=in

If missing: netsh advfirewall firewall add rule name="Jellyfin" dir=in action=allow protocol=TCP localport=8096

🔐 Add the tv.ixo.lol block to Caddyfile

See the Caddyfile example in Step 3 above. The tv.ixo.lol block proxies to Jellyfin without the forward_auth wrapper. Reload Caddy after saving:

caddy reload --config
Tell TV app users their server address

When Jeremy sets up the Jellyfin app on his LG TV:

Server address:  https://tv.ixo.lol
Username:        jeremy          ← his Jellyfin-native username
Password:        [set by admin]  ← NOT his Google password

After first login the app saves these — he won't need to re-enter them unless he reinstalls.

🔄 Step 5 — DDNS for tv.ixo.lol (Home IP Changes)

Your home internet IP changes occasionally. When it does, tv.ixo.lol stops working until the DNS record is updated. This script runs every 5 minutes as a scheduled task and updates the Porkbun DNS record automatically.

Save as \ddns_porkbun.ps1:

# ddns_porkbun.ps1
# Requires a Porkbun API key pair — create at porkbun.com/account/api
$API_KEY        = "pk1_YOUR_API_KEY_HERE"
$SECRET_API_KEY = "sk1_YOUR_SECRET_KEY_HERE"
$DOMAIN         = "ixo.lol"
$SUBDOMAIN      = "tv"

# Get current public IP
$currentIP = (Invoke-WebRequest -Uri "https://api.ipify.org" -UseBasicParsing).Content.Trim()

# Get existing DNS record
$existing = Invoke-RestMethod `
    -Uri "https://porkbun.com/api/json/v3/dns/retrieveByNameType/$DOMAIN/A/$SUBDOMAIN" `
    -Method POST `
    -ContentType "application/json" `
    -Body (@{ apikey = $API_KEY; secretapikey = $SECRET_API_KEY } | ConvertTo-Json)

$recordIP = $existing.records[0].content

if ($currentIP -ne $recordIP) {
    $recordID = $existing.records[0].id
    Invoke-RestMethod `
        -Uri "https://porkbun.com/api/json/v3/dns/edit/$DOMAIN/$recordID" `
        -Method POST `
        -ContentType "application/json" `
        -Body (@{
            apikey       = $API_KEY
            secretapikey = $SECRET_API_KEY
            name         = $SUBDOMAIN
            type         = "A"
            content      = $currentIP
            ttl          = "60"
        } | ConvertTo-Json)
    Write-Host "$(Get-Date) — Updated tv.ixo.lol: $recordIP → $currentIP"
} else {
    Write-Host "$(Get-Date) — No change: tv.ixo.lol = $currentIP"
}
Schedule as a Windows task
# Run once to register the scheduled task (PowerShell as Admin):
$action  = New-ScheduledTaskAction -Execute "powershell.exe" `
           -Argument "-NonInteractive -File D:\Server\ddns\ddns_porkbun.ps1"
$trigger = New-ScheduledTaskTrigger -RepetitionInterval (New-TimeSpan -Minutes 5) `
           -Once -At (Get-Date)
$settings = New-ScheduledTaskSettingsSet -ExecutionTimeLimit (New-TimeSpan -Minutes 1)
Register-ScheduledTask -TaskName "DDNS-tv.ixo.lol" `
    -Action $action -Trigger $trigger -Settings $settings `
    -RunLevel Highest -Force

# Confirm it's registered:
Get-ScheduledTask -TaskName "DDNS-tv.ixo.lol"

💡
After a friend installs Tailscale and logs in with their Google account, you need to approve their device in the Tailscale admin panel → Machines → Approve. Until approved, they can't reach the server. This is the access control gate — don't approve devices you don't recognise.
🔐 Send the invite (Tailscale admin panel)

Tailscale admin → Users → Invite User → enter their email. They get a link that adds them to your Tailnet after they install the app and sign in.

Alternatively, share your Tailnet name () and tell them to join at tailscale.com — they'll be prompted to join your network on login.

🔐 Approve the device

Admin panel → Machines. The new device appears as pending. Review it, confirm it's theirs, approve it. Once approved, they can reach and any other service you've given them access to via Authentik.

Tell them how to connect

Once on the Tailnet, they open a browser and go to . They click the Google sign-in, Authentik validates them, and they land on the server homepage. Tailscale must be running on their device — they'll see it in the system tray or phone status bar.

🖥️ Adding a Friend's Machine as a Service Node

A friend can host services (Minecraft, a second Sunshine instance, etc.) on their own machine. Your server acts as the hub — users always connect to your server, which proxies to their machine over Tailscale. Users never need to know their friend's IP.

If the friend's machine goes offline, any service routed through it shows as unavailable on the dashboard with a message identifying whose machine it is (e.g. "Dave's PC is currently offline").

Friend installs Tailscale and joins the Tailnet

Same process as above. After their machine is approved, note its Tailscale IP from the admin panel.

🔐 Update nodes.json with the new node

Fill in one of the stub entries (node-a, node-b) in :

"id":                "node-a",
"displayName":       "Dave's Gaming PC",
"owner":             "dave",               ← must match Authentik username
"tailscaleIP":       "100.x.x.x",          ← from Tailscale admin panel
"tailscaleHostname": "daves-pc.ixo.ts.net",
"services":          ["minecraft-vanilla"], ← service IDs from services.json
"healthCheckPort":   25565,
"offlineMessage":    "Dave's PC is currently offline — Minecraft is unavailable."
🔐 Add a Caddy reverse proxy entry pointing at their Tailscale IP 
# In Caddyfile — proxy Minecraft (TCP passthrough, not HTTP):
# For game servers, use Caddy's layer4 plugin or a direct port forward
# from the hub to the friend node's Tailscale IP.

# For HTTP services on a friend node:
friendservice.ixo.lol {
    forward_auth localhost:9000 {
        uri /outpost.goauthentik.io/auth/caddy
        copy_headers X-authentik-username X-authentik-groups X-authentik-email
        trusted_proxies private_ranges
    }
    reverse_proxy 100.x.x.x:PORT  ← friend node's Tailscale IP:port
}
🔐 Add the service to services.json

Add an entry to services.json with the service details. The dashboard will automatically pick it up, poll the friend node's health check port, and display the red/green status dot with the owner attribution.

⚠️
The tv.ixo.lol DNS record contains your real home IP. This is unavoidable for Smart TV support. Anyone who looks up tv.ixo.lol can see your home IP. This is an acceptable tradeoff for a personal server — it only enables access to Jellyfin's login page, which requires a valid username and password to proceed.
Certificate transparency and subdomain enumeration

When Caddy requests TLS certificates for *.ixo.lol, Let's Encrypt logs this to public certificate transparency logs (visible at crt.sh). Anyone can see that ixo.lol has a wildcard certificate. They cannot see which subdomains are actually in use, and they cannot connect to them — all non-TV subdomains resolve to a Tailscale IP which is unreachable without being on the Tailnet. This is not a meaningful security concern for a personal server.
