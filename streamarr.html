📡

Streamarr

// STRM Virtual Library · Port · Python / Docker WSL2 · LAN Only //

Streamarr generates lightweight .strm stub files that point Jellyfin at streaming URLs for very long or very popular shows (Pokémon, One Piece, Naruto, etc.) that aren't worth hundreds of gigabytes of local storage. It runs inside the seedbox's Mullvad VPN container so all outbound streaming traffic exits through the tunnel. A watch-rule engine monitors TMDB/TVDB signals and automatically mints new stubs when matching content is released.

First Time Setup Complete
Mark as done → activates Overview tab
💪
Pi Sentry Compatibility PC ONLY

Cannot run on the Pi Sentry — must stay on the Windows PC.

Streamarr runs inside a Mullvad VPN network namespace (Gluetun in WSL2) which is tightly coupled to the Windows/WSL2 environment on the PC. Replicating the Gluetun namespace setup on Pi OS is possible but adds significant complexity with no clear benefit.

🐉 V1 Goal: Streamarr container running inside Mullvad namespace → at least one virtual show library scanning in Jellyfin → one watch rule firing on a TMDB keyword trigger.
🗺️ How It Fits Into the Seedbox Stack

Streamarr is not a download manager — it never stores video. It creates .strm files: one-line text files containing a streaming URL (e.g. a direct link from an yt-dlp-resolvable source). Jellyfin treats each .strm as a playable media item and fetches the stream on demand. All outbound requests from Streamarr route through the Mullvad tunnel, matching the existing seedbox VPN policy.

⚙️ Step 1 — Create the Docker Compose Service (WSL2)
🔐 Admin only — run inside WSL2 Ubuntu

Streamarr runs as a Docker container in the same WSL2 instance as Authentik and Immich. It shares the mullvad_net network namespace so all outbound traffic exits through the VPN tunnel.

Create the project directory 
mkdir -p ~/streamarr/config ~/streamarr/rules ~/streamarr/logs
cd ~/streamarr
Create docker-compose.yml 
version: "3.9"
services:
  streamarr:
    image: python:3.12-slim
    container_name: streamarr
    restart: unless-stopped
    # ── Route all traffic through Mullvad tunnel ──────────────────
    # Option A: network_mode shares the VPN container's namespace.
    # Replace 'gluetun' with whatever your VPN sidecar container is named
    # if you run one. If you bind qBittorrent via Mullvad's Windows
    # adapter instead, use Option B below.
    #
    # Option A (recommended — VPN sidecar like Gluetun in WSL2):
    network_mode: "service:gluetun"
    #
    # Option B (no VPN sidecar — host network, Mullvad bound on Windows):
    # network_mode: host
    # ─────────────────────────────────────────────────────────────
    volumes:
      - ./config:/app/config          # streamarr.yml + rules/
      - ./rules:/app/rules            # watch rule YAML files
      - ./logs:/app/logs
      # Mount the Virtual library path where Jellyfin can see it.
      # This is a Windows path exposed into WSL2 via /mnt/d/
      - /mnt/d/Media/Virtual:/strm-output
    ports:
      # Only expose to LAN — Caddy is NOT proxying this (admin-only tool)
      - "127.0.0.1::"
    environment:
      - STREAMARR_CONFIG=/app/config/streamarr.yml
      - STREAMARR_OUTPUT=/strm-output
      - TMDB_API_KEY=${TMDB_API_KEY}
      - TVDB_API_KEY=${TVDB_API_KEY}
      - TZ=America/New_York
    working_dir: /app
    command: >
      bash -c "pip install -q flask requests pyyaml schedule yt-dlp &&
               python streamarr.py"
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:/health"]
      interval: 60s
      timeout: 10s
      retries: 3
💡
No Gluetun yet? If you're currently binding qBittorrent to the Windows Mullvad adapter rather than a Docker VPN sidecar, use network_mode: host for now and revisit adding a Gluetun container to WSL2 later (it gives you proper per-container VPN isolation without touching the Windows adapter config).
Create the .env file with your API keys 
# ~/streamarr/.env
TMDB_API_KEY=your_tmdb_v3_api_key_here
TVDB_API_KEY=your_tvdb_api_key_here

Get TMDB key free at themoviedb.org/settings/api. TVDB key at thetvdb.com/dashboard/account/apikeys.

📝 Step 2 — Write streamarr.yml
Create ~/streamarr/config/streamarr.yml 
server:
  port: 
  host: 0.0.0.0

output:
  base_path: /strm-output        # maps to D:\Media\Virtual\ on Windows
  # Subdirectory structure mirrors Jellyfin's expected TV/movie layout
  tv_subdir:     TV              # → D:\Media\Virtual\TV\{show}\Season {N}\{ep}.strm
  movie_subdir:  Movies          # → D:\Media\Virtual\Movies\{title} ({year})\{title}.strm

jellyfin:
  # Streamarr calls Jellyfin's API to trigger library scans after writing stubs
  url:    http://host.docker.internal:
  api_key: ""                    # generate in Jellyfin: Dashboard → API Keys

sources:
  # Resolvers tell Streamarr how to build a streaming URL for a given show.
  # 'yt-dlp' uses yt-dlp to extract a direct stream URL at playback time.
  # 'static'  uses a hardcoded URL template (e.g. an Invidious instance).
  default_resolver: yt-dlp
  yt_dlp_opts:
    format: "bestvideo[height<=1080]+bestaudio/best[height<=1080]"
    # Rate-limit to be polite to source; all goes through VPN anyway
    ratelimit: "5M"

  # Optional: use your local Invidious as a YouTube proxy instead of yt-dlp
  # invidious_base: "http://localhost:"

watcher:
  # How often the rule engine polls TMDB/TVDB for new episodes/releases
  poll_interval_hours: 12
  # Run rules scan on startup in addition to schedule
  run_on_start: true

logging:
  path: /app/logs/streamarr.log
  level: INFO
  rotate_days: 14
📺 Step 3 — Define Virtual Shows (Manual STRM Generation)

Each show you want to virtualize gets a YAML rule file in ~/streamarr/rules/. Streamarr reads these on startup and generates one .strm file per episode. New episodes are minted automatically on the next watcher poll if they match the episode range.

Example: rules/pokemon.yml 
id: pokemon-og
type: tv
name: "Pokémon"
tmdb_id: 60572              # confirms metadata; also drives poster/synopsis fetch
tvdb_id: 73796
resolver: yt-dlp
source_query: "Pokémon {season} {episode} official English dub"
# Generate stubs for these seasons/episodes.
# Use 'all' to generate for every episode TMDB knows about.
# Use explicit ranges to stay conservative on stub count.
seasons:
  - season: 1
    episodes: all           # all 52 of Season 1
  - season: 2
    episodes: all
  - season: 3
    episodes: all           # etc. — add seasons as you want them in Jellyfin

# Optional: freeze at a specific episode if you don't want the full run
# max_episode: 200

# Tags written into Jellyfin NFO alongside the stub
tags:
  - anime
  - pokemon
  - nostalgic
notes: "Classic series. Not worth 800 GB. Watch on demand via VPN."
Example: rules/onepiece.yml 
id: one-piece
type: tv
name: "One Piece"
tmdb_id: 37854
tvdb_id: 81797
resolver: yt-dlp
source_query: "One Piece Episode {absolute_episode} English Sub"
seasons:
  - season: 1
    episodes: all           # Toei's S1 numbering (episodes 1-61)
  # Add more seasons as desired — One Piece has ~20 Toei seasons
max_episode: 1100           # cap stub generation; extend when you catch up
Trigger a manual stub generation run 
# On WSL2 host — fires the generator against all rule files immediately
curl -X POST http://localhost:/api/generate/all

# Or generate a single show by its rule ID:
curl -X POST http://localhost:/api/generate/pokemon-og

# Watch output:
tail -f ~/streamarr/logs/streamarr.log

After this runs, D:\Media\Virtual\TV\Pokémon\Season 01\Pokémon S01E01.strm etc. will exist. Add D:\Media\Virtual\TV as a TV Shows library in Jellyfin — it'll scan and pick them up within 30 minutes (or trigger a manual scan).

# In streamarr.yml, under sources:
url_cache:

🔐 Step 6 — SSO / Access Control
No external SSO — admin-only LAN tool

Streamarr's web UI (http://localhost:) is bound to 127.0.0.1 only and is never proxied through Caddy. It is not exposed to users. All user-facing access to virtual shows is through Jellyfin as usual — Streamarr is invisible to them.

Protect the Streamarr API with a static bearer token set in streamarr.yml:

# In streamarr.yml → server:
api_key: "generate-a-strong-random-token-and-store-in-vaultwarden"

All API calls then require: Authorization: Bearer <token>

📡
Streamarr keeps infinite-run shows in Jellyfin without burning storage. A Pokémon library with 1,000+ episodes costs < 1 MB on disk (one tiny .strm file per episode). The VPN ensures streaming traffic is private.

🔍 Watch Rule Engine

The watcher polls TMDB and TVDB every 12 hours (configurable). For any rule marked auto_extend: true, it automatically generates new stubs when new episodes appear. Rules can also trigger on broader signals — franchise keywords, director credits, production companies, etc.

Rule Type · Show Series
Auto-extend running series
When TMDB reports a new episode for a tracked show ID, Streamarr mints the .strm stub and triggers a Jellyfin library scan.
auto_extend: true
tmdb_id: 60572
Rule Type · Franchise / Keyword
Keyword-triggered mock entries
Monitor a TMDB keyword (e.g. "pokemon" keyword ID 9643). Any new movie or show tagged with it gets a stub entry — good for catching new Pokémon movies or spin-offs automatically.
type: keyword_watch
tmdb_keyword_id: 9643
stub_type: movie
Rule Type · Director
Director watch
Any new TMDB credit matching a given director ID triggers a stub. Good for following auteur directors whose complete output you want visible without storing it.
type: person_watch
tmdb_person_id: 138
role: director
Rule Type · Actor
Actor / cast watch
Monitor a cast member's TMDB filmography. New theatrical releases matching minimum vote threshold get stubs added to the virtual Movies library.
type: person_watch
tmdb_person_id: 1136406
role: actor
min_vote_count: 50
Rule Type · Production Company
Studio output watch
Watch all new releases from a production company (e.g. A24, Studio Ghibli). Every new theatrical release with a valid TMDB entry gets a stub.
type: company_watch
tmdb_company_id: 21
# 21 = Lucasfilm
Rule Type · Collection
Movie collection / saga
Monitor a TMDB Collection (e.g. the MCU collection). Any new entry in the collection gets a stub. Perfect for long film sagas where you want stubs but not 50 GB of storage.
type: collection_watch
tmdb_collection_id: 131296
# Dragon Ball films
Rule Type · Network / Channel
Network output watch
Monitor a broadcast network or streaming service. All new shows from a TVDB network ID above a vote threshold get tracked. Good for following a specific anime studio's seasonal releases.
type: network_watch
tvdb_network_id: 349
# 349 = Cartoon Network
min_episodes: 10
Rule Type · Manual Override
One-off static stub
Write a stub for a specific URL you already have — bypasses the resolver engine entirely. Useful for content not indexable by yt-dlp.
type: static
name: "Show Name"
url: "https://direct-url"

🔄 Updating yt-dlp

yt-dlp extractors break regularly when platforms update their front-ends. Update yt-dlp inside the running container:

# Quick update (persists until container rebuild):
docker exec streamarr pip install -U yt-dlp

# Permanent: rebuild the image so the updated version is baked in:
cd ~/streamarr
docker-compose down && docker-compose up -d --build
🧹 Purging Dead Stubs

Dead stubs are .strm files whose URLs can no longer be resolved. Streamarr can validate the full library and remove or flag them:

# List dead stubs (dry run):
curl http://localhost:/api/stubs/dead \
  -H "Authorization: Bearer YOUR_API_KEY"

# Remove dead stubs (irreversible — they'll be re-created on next
# generate run if the source comes back):
curl -X DELETE http://localhost:/api/stubs/dead \
  -H "Authorization: Bearer YOUR_API_KEY"

🔴 Playback fails: "Unable to connect to stream"
yt-dlp failed to resolve the source URL. Check in order:
  • Is Streamarr running? docker ps | grep streamarr
  • Is Mullvad/Gluetun up? curl http://localhost:/api/vpn/status
  • Is yt-dlp outdated? docker exec streamarr yt-dlp --version — update if > 2 weeks old
  • Is the source query finding anything? Test manually: docker exec streamarr yt-dlp --get-url "ytsearch1:Pokémon S01E01 English dub"
  • Check logs: docker logs streamarr --tail 50
⚠️ Virtual shows don't appear in Jellyfin after generation
  • Verify .strm files exist: ls "D:\Media\Virtual\TV\Pokémon\Season 01\"
  • Trigger a manual Jellyfin library scan: Dashboard → Libraries → Virtual TV → Scan Library
  • Check folder naming — Jellyfin requires Show Name\Season NN\Show Name SXXEXX.strm
  • Confirm Jellyfin has read access to D:\Media\Virtual\
⚠️ Watch rules not firing automatically
  • Check watcher.enabled: true in streamarr.yml
  • Verify TMDB/TVDB API keys are set: docker exec streamarr env | grep API_KEY
  • Check the log for watcher cycle entries: grep "watcher" ~/streamarr/logs/streamarr.log
  • Test manually: curl -X POST http://localhost:/api/watcher/run -H "Authorization: Bearer YOUR_API_KEY"
⚠️ VPN check fails — traffic may not be tunneled
  • If using Gluetun: docker inspect streamarr | grep NetworkMode — should show container:gluetun
  • If using host network mode: verify Mullvad Windows client is connected before Streamarr starts
  • IP leak test from inside the container: docker exec streamarr curl https://am.i.mullvad.net/json — should show a Mullvad IP
💡 Playback is slow / buffers on first play
Expected behaviour on first play — yt-dlp is resolving the URL live. Enable the URL cache (url_cache.enabled: true, ttl_hours: 4) in streamarr.yml to cache resolved URLs. Subsequent plays within the TTL window will be instant. Note that streaming URLs typically expire within a few hours, so don't set TTL above 4–6 hours.
💡 Metadata / poster is wrong for a virtual show
Streamarr writes .nfo sidecars fetched from TMDB. If the poster is wrong: delete the .jpg file and the .nfo file for that show's folder (not the .strm files), then call POST /api/metadata/refresh/{rule-id}. Jellyfin will re-read the corrected NFOs after the next scan.

🗺️ What "Adding a Show" Actually Means

Streamarr doesn't have a GUI "Add Show" button. Each virtual show is a YAML rule file in ~/streamarr/rules/. When you run the generator, Streamarr reads that file, queries TMDB/TVDB for episode metadata, and writes one .strm + .nfo + .jpg per episode into D:\Media\Virtual\TV\. Jellyfin picks those files up on the next scan — no video is ever downloaded or stored.

🔢 Step 1 — Get the TMDB and TVDB IDs
Find the TMDB ID

Search at themoviedb.org. The ID is in the URL: themoviedb.org/tv/60572-pokemon.

Find the TVDB ID

Search at thetvdb.com. The ID appears in the right panel under "TheTVDB.com Series ID", or in the URL: thetvdb.com/series/73796.

💡
For long-running anime with no consistent season numbering (One Piece, Naruto, Dragon Ball Z), use {absolute_episode} in source_query instead of {season}/{episode} — this matches how uploads are titled on source sites.
📝 Step 2 — Write the Rule File
Create ~/streamarr/rules/[showname].yml

Minimal TV series rule:

id:           dragon-ball-z
type:         tv
name:         "Dragon Ball Z"
tmdb_id:      12971
tvdb_id:      76107
resolver:     yt-dlp
source_query: "Dragon Ball Z Episode {absolute_episode} English Dub"
auto_extend:  true     # auto-mint stubs when TMDB reports new episodes
enabled:      true

seasons:
  - season: 1
    episodes: all
  - season: 2
    episodes: all
  # add more seasons as needed, or cap with:
  # max_episode: 291

tags:
  - anime
  - virtual

For a movie, set type: movie and omit seasons. Streamarr generates one stub + NFO for the film.

For a keyword watch (auto-catch new releases by TMDB tag) 
id:               watch-pokemon-movies
type:             keyword_watch
name:             "Pokémon Movies (auto)"
tmdb_keyword_id:  9643
stub_type:        movie
resolver:         yt-dlp
auto_extend:      true
min_vote_count:   25
enabled:          true

Silently mints a stub whenever TMDB indexes a new title tagged with that keyword. No further manual action needed after initial setup.

⚡ Step 3 — Trigger Stub Generation
Generate stubs for the new rule only 
# Replace the id with the one from your rule file
curl -X POST http://localhost:/api/generate/dragon-ball-z   -H "Authorization: Bearer YOUR_API_KEY"

# Watch the log to see stubs being written:
tail -f ~/streamarr/logs/streamarr.log

Streamarr fetches episode metadata from TMDB/TVDB, writes .strm + .nfo + .jpg files into D:\Media\Virtual\TV\Dragon Ball Z\Season 01\ etc., then calls Jellyfin's scan API automatically (if jellyfin.api_key is set in streamarr.yml).

Or run all rules at once 
curl -X POST http://localhost:/api/generate/all   -H "Authorization: Bearer YOUR_API_KEY"

Safe to re-run at any time — existing stubs are not overwritten unless you pass force=true.

✅ Step 4 — Verify in Jellyfin
Check the Virtual TV library

The show should appear in the Jellyfin grid with poster art and an episode list. If it's not there yet, trigger a manual scan — see the Jellyfin Rescan tab.

Test playback

Click any episode → Play. Jellyfin reads the .strm, hands the URL to yt-dlp, and streams it. Expect a ~2–4s delay on first play while the URL is resolved. Enable URL caching in streamarr.yml for instant repeat plays:

# In streamarr.yml → sources:
url_cache:
  enabled: true
  ttl_hours: 4    # streaming URLs rotate; don't set higher than 4–6
🗑️ Removing a Virtual Show

Set enabled: false in the rule file (or delete it), then delete the show folder from D:\Media\Virtual\TV\[Show Name]\. Trigger a Jellyfin rescan — Jellyfin will notice the folder is gone and drop the entry from the library automatically. No other shows are affected.

⚠️
Never use "Replace All Metadata" or "Refresh All Metadata." These wipe existing posters and episode data for every item in the library and re-fetch everything from scratch — triggering hundreds of TMDB/TVDB requests and potentially rate-limiting the server. Only the targeted methods below are safe.
✅ The Automatic Path — Streamarr Notifies Jellyfin

After every successful stub generation run, Streamarr calls Jellyfin's scan API directly — no manual action needed — as long as the API key is set in streamarr.yml:

# ~/streamarr/config/streamarr.yml
jellyfin:
  url:     http://host.docker.internal:
  api_key: "your-jellyfin-api-key-here"   # ← must be populated

Generate the key at Jellyfin Dashboard → API Keys → + New Key (label it "streamarr"). With this in place, the Virtual TV library updates automatically every time you run /api/generate/.

🔧 Manual Scan — When You Actually Need It

Use a manual scan when: stubs were written but Jellyfin hasn't picked them up, the Streamarr → Jellyfin auto-notify failed (check the log), or you edited stub files by hand outside of the generator.

Option A — Scan the Virtual library only (preferred)

Jellyfin Dashboard → Libraries → hover "Virtual TV" → click ⋮ → Scan Library.

This does a file-system diff: new .strm files get added, removed files get dropped. No metadata is re-fetched for existing items. Only brand-new stubs trigger a metadata read (from their accompanying .nfo — not a live API call).

Option B — Scan via Jellyfin API (scriptable) 
# New-content-only scan — safe, won't touch existing metadata
curl -X POST   "http://localhost:/Library/Refresh?refreshtype=NewContent"   -H "X-Emby-Token: YOUR_JELLYFIN_API_KEY"
💡
refreshtype=NewContent is the safe flag — finds new files only, skips re-fetching anything already in the database.
Option C — Fix metadata on one show (most surgical)

Navigate to the show in Jellyfin → ⋮ → Refresh Metadata → select "Search for missing metadata and images only" (not "Replace all metadata"). Fills in any gaps without touching what's already correct.

🚫 What Not to Do
ActionWhy it's bad
Dashboard → Scan All Libraries Scans every library at once — unnecessary if you only added stubs to Virtual TV
⋮ → Refresh Metadata → "Replace all metadata" Wipes all posters, synopses, and episode data then re-downloads everything. Never needed unless metadata is systemically corrupted.
Restarting Jellyfin as a scan trigger Doesn't trigger a scan. Just causes a 30–60s outage for anyone currently streaming.
Deleting metadata\ folders manually Forces a full re-fetch for every item in the library. Last resort only.
🗓️ Scheduled Scans

Jellyfin has a built-in scheduled task that scans all libraries automatically. Check it at Dashboard → Scheduled Tasks → "Scan Media Library". Recommended: nightly at 3–5am — after Streamarr's 12h watcher polls have had time to finish, but before anyone is actively streaming. Acts as a safety net for anything the Streamarr auto-notify may have missed.
