Flash Arcade

// Ruffle WASM · .swf Preservation · Per-User .sol Saves · Authentik Forward Auth · nginx · Docker //

A private browser-based Flash game arcade powered by Ruffle (WebAssembly Flash emulator). Games load from a shared read-only library; save files (.sol) are stored per-user under userdata/{NAME}/arcade/saves/. Authentication is handled by Authentik via the X-authentik-username header — no separate login screen.

🐉 V1 Goal: Serve .swf games via Ruffle in the browser with per-user .sol save file isolation. No Flash plugin required — Ruffle runs entirely in WebAssembly.
📁 Step 1 — Create Directory Structure

Create the shared game library and a saves folder for each user. The common library is read-only for all users; only the saves directory is writable and per-user.

# Shared game library (read-only, all users)
mkdir -p userdata/common/arcade/bin
mkdir -p userdata/common/arcade/library

# Per-user saves — repeat for each user (NAME = Authentik username)
mkdir -p userdata/{NAME}/arcade/saves
🐉
Naming rule: {NAME} must exactly match the Authentik username. Caddy injects it via X-authentik-username. Mismatches = wrong saves directory.
⚙️ Step 2 — Install Ruffle Runtime WASM

Download the latest self-hosted build from ruffle.rs and extract the files into userdata/common/arcade/bin/.

userdata/common/arcade/bin/
  ruffle.js
  ruffle_bg.wasm        ← must be served as application/wasm
  …
⚠️
MIME type is mandatory: The .wasm file must be served with Content-Type: application/wasm. Browsers silently refuse to load it otherwise. Add this to your nginx config inside the arcade container's server {} block:
location ~* \.wasm$ {
    types { }
    default_type application/wasm;
    add_header Cache-Control "public, max-age=86400";
}
🎮 Step 3 — Add Games to the Library

Drop .swf files into userdata/common/arcade/library/ and register them in library.json. No service restart needed — nginx serves static files directly.

userdata/common/arcade/library/
  game1.swf
  game1.png             ← thumbnail (optional)
  library.json
# library.json format:
[
  {
    "id":    "game1",
    "title": "My Flash Game",
    "swf":   "library/game1.swf",
    "thumb": "library/game1.png"
  }
]
🐳 Step 4 — Docker Compose DOCKER
docker-compose.yml — arcade service block
services:
  arcade:
    image: nginx:alpine
    volumes:
      - ./userdata/common/arcade:/usr/share/nginx/html:ro
      - ./userdata/${USER}/arcade/saves:/usr/share/nginx/html/saves
    environment:
      - AUTHENTIK_USER=${USER}
    restart: unless-stopped

The ${USER} variable is resolved from the environment at container start. The shared library is mounted read-only; only the saves/ subdirectory is writable. When a different user logs in, restart the container (or re-up) with the new USER value to re-point the saves mount.

⚠️
Single-seat note: The standard Docker approach mounts one user's saves at a time. For true multi-user concurrency, run one container instance per user on separate ports, each with their own USER value, and route via Caddy subdomain or path prefix.
🔐 Step 5 — Authentik Forward Auth (SSO) SSO REQUIRED

Caddy protects the arcade via Authentik forward auth. Every request is authenticated before reaching nginx. Authentik injects X-authentik-username, which routes save files to the correct user directory.

# Caddyfile entry:
. {
    forward_auth localhost:9000 {
        uri /outpost.goauthentik.io/auth/caddy
        copy_headers X-authentik-username
        trusted_proxies private_ranges
    }
    reverse_proxy localhost:
}

The save server reads X-authentik-username from the header to determine which user's saves directory to read/write.

# Save routing logic:
# PUT /saves/game_name.sol
#   → username = request.headers.get('X-authentik-username')
#   → write to: userdata/{username}/arcade/saves/game_name.sol

# GET /saves/game_name.sol
#   → username = request.headers.get('X-authentik-username')
#   → read from: userdata/{username}/arcade/saves/game_name.sol
🖥️ Step 6 — Embed Ruffle in the Game Player Page
<!-- In your game player page -->
<script>
  window.RufflePlayer = window.RufflePlayer || {};
  window.RufflePlayer.config = {
    autoplay: "on",
    unmuteOverlay: "hidden",
  };
</script>
<script src="/bin/ruffle.js"></script>
<script>
  window.addEventListener("load", () => {
    const ruffle = window.RufflePlayer.newest();
    const player = ruffle.createPlayer();
    document.getElementById("game-container").appendChild(player);
    player.load("/library/game1.swf");

    // Per-user save hooks
    player.addEventListener("savegame", async (e) => {
      await fetch(`/saves/${e.detail.name}.sol`, {
        method: 'PUT',
        body: e.detail.data,
        headers: { 'Content-Type': 'application/octet-stream' }
      });
    });

    player.addEventListener("loadgame", async (e) => {
      const res = await fetch(`/saves/${e.detail.name}.sol`);
      if (res.ok) e.detail.resolve(await res.arrayBuffer());
      else e.detail.resolve(null);
    });
  });
</script>
<div id="game-container"></div>

Ruffle's savegame / loadgame events intercept Flash SharedObject writes and proxy them to /saves/, which nginx serves from the user's personal saves directory.

🗺️ Architecture Overview

The arcade has two layers: a shared read-only game library (common to all users) and a per-user writable saves directory. nginx serves both. The correct saves directory is mounted based on the authenticated Authentik username injected by Caddy at the proxy layer.

🐉
Core principle: The game library is identical for every user. Only the saves mount changes. Caddy reads X-authentik-username from Authentik and uses it to mount the correct user saves path before nginx serves anything.
Request → Caddy → Authentik forward auth
                        ↓ X-authentik-username: alice
              nginx (arcade container)
                ├── /library/   → userdata/common/arcade/library/  (read-only, shared)
                ├── /bin/       → userdata/common/arcade/bin/       (Ruffle WASM)
                └── /saves/     → userdata/alice/arcade/saves/      (writable, per-user)

When provisioning a new user, create their arcade saves directory. The session manager's ensure_user_dirs() does this automatically on first login, but you can pre-create it manually:

💾 User Data Storage → SSO TAB

Save files are stored at userdata/{NAME}/arcade/saves/, where NAME is the Authentik username injected by Caddy. Each user's saves are completely isolated. This path follows the same per-user convention as other services on the server.

userdata/
  common/
    arcade/
      bin/              ← Ruffle WASM runtime (shared, read-only)
      library/          ← All .swf game files (shared, read-only)
        game1.swf
        library.json
  alice/
    arcade/
      saves/
        game1.sol       ← Alice's save
        game2.sol
  bob/
    arcade/
      saves/
        game1.sol       ← Bob's separate save — never touches Alice's

See the SSO tab → User Data Storage for the full cross-service user data architecture.

🔁 Backup

The game library (userdata/common/arcade/) should be included in the server backup — especially library.json and any custom thumbnails. The bin/ folder contains Ruffle and can be re-downloaded, but backing it up avoids version surprises.

Per-user saves/ directories are covered by the standard per-user backup that runs nightly for all userdata/{NAME}/ folders.

🛠️ Common Issues
Game doesn't load — browser shows a blank area or spinner forever
Open DevTools → Network. Check if ruffle_bg.wasm is being fetched. If it returns 404, the bin/ directory isn't mounted correctly. If it returns 200 but the game still doesn't start, check the Console tab for Ruffle errors — the game may use unsupported ActionScript.
.wasm file loads but browser refuses to run it
The application/wasm MIME type is missing. Check nginx config: location ~* \.wasm$ { default_type application/wasm; }. The browser console will say something like "Failed to load WebAssembly: incorrect MIME type". This is the most common first-run issue.
Saves don't persist between sessions
Verify the saves/ volume is mounted writable (no :ro flag). Confirm the PUT /saves/*.sol request succeeds in DevTools → Network. If it returns 403, the saves directory may not be writable by the nginx user inside the container.
User can see another user's saves
The ${USER} env variable was not set, or the container wasn't restarted after changing users. Check that Caddyfile has copy_headers X-authentik-username and that the save routing logic reads the header. This is a data isolation failure — confirm and fix before letting users play.
Game library page is empty / shows no games
Check that library.json is valid JSON and is in the correct location (userdata/common/arcade/library/library.json). The frontend reads this file to build the game list. Use python3 -m json.tool library.json to validate.
403 or redirect loop when accessing the arcade
Authentik forward auth is blocking the request. Check the Authentik admin panel to confirm the outpost is running and the arcade application is configured correctly. Also confirm Caddy's trusted_proxies private_ranges is set — without it, Caddy may reject the forward auth callback.
Game works but audio is muted by default
This is expected Ruffle behavior — browsers require a user interaction before allowing audio autoplay. The unmuteOverlay: "hidden" config hides the overlay but the user still needs to click the game area once to unmute. This is a browser security restriction, not a bug.
