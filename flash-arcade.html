Flash Arcade

// Ruffle · .swf Preservation · Per-User Saves · Authentik SSO · nginx · Docker //

A private browser-based Flash game arcade powered by Ruffle (WebAssembly Flash emulator). Games load from a shared read-only library; save files (.sol) are stored per-user. Authentication is handled entirely by Authentik via the X-authentik-username header — no separate login needed.

🐉 V1 Goal: Serve .swf games via Ruffle in the browser with per-user .sol save file isolation. No Flash plugin required — Ruffle runs entirely in WebAssembly.
🗺️ Architecture Overview

The arcade has two layers: a shared read-only game library (common to all users) and a per-user writable saves directory. nginx serves both. The correct saves directory is mounted based on the authenticated Authentik username injected by Caddy at the proxy layer.

🐉
Core principle: The game library is identical for every user. Only the saves mount changes. Caddy reads X-authentik-username from Authentik and passes it as an environment variable — the container mounts the correct saves path before serving.
🗂️ Directory Structure
userdata/
  common/
    arcade/
      bin/              ← Ruffle WASM runtime files (ruffle.js, ruffle_bg.wasm, …)
      library/          ← All .swf game files (shared, read-only for all users)
        game1.swf
        game2.swf
        library.json    ← Game manifest (id, title, thumbnail, filename)
  {NAME}/               ← Per-user directory — NAME = Authentik username
    arcade/
      saves/            ← User's .sol save files, one per game

The common/arcade/ directory is mounted read-only and is identical for every user. The userdata/{NAME}/arcade/saves/ directory is writable and unique per user. Create it when provisioning a new user:

mkdir -p userdata/{NAME}/arcade/saves
🐳 Docker Compose
docker-compose.yml — arcade service block
services:
  arcade:
    image: nginx:alpine
    volumes:
      - ./userdata/common/arcade:/usr/share/nginx/html:ro
      - ./userdata/${USER}/arcade/saves:/usr/share/nginx/html/saves
    environment:
      - AUTHENTIK_USER=${USER}
    restart: unless-stopped

The ${USER} variable is resolved from the environment at container start. The shared library is read-only; only the saves/ subdirectory is writable. When a different user logs in, restart the container (or re-up) with the new USER value to re-point the saves mount.

⚠️
Single-seat note: The standard Docker approach mounts one user's saves at a time. For true multi-user concurrency (multiple users playing simultaneously), run one container instance per user on separate ports, each with their own USER value, and route via Caddy subdomain or path prefix.
🔐 Authentication — Authentik Forward Auth

Caddy protects the arcade via Authentik forward auth. Every request is validated before reaching nginx. Authentik injects X-authentik-username, which is used to route save files to the correct user directory.

# Caddyfile entry:
arcade.yourdomain.com {
    forward_auth localhost:9000 {
        uri /outpost.goauthentik.io/auth/caddy
        copy_headers X-authentik-username
        trusted_proxies private_ranges
    }
    reverse_proxy localhost:[ARCADE_PORT]
}

The save server reads the username from the header to determine which user's saves directory to read/write:

# Save routing logic (in save handler):
# POST/PUT /saves/game_name.sol
#   → username = request.headers.get('X-authentik-username')
#   → write to: userdata/{username}/arcade/saves/game_name.sol

# GET /saves/game_name.sol
#   → username = request.headers.get('X-authentik-username')
#   → read from: userdata/{username}/arcade/saves/game_name.sol
⚙️ Ruffle Runtime Setup
Installing Ruffle

Ruffle is a WebAssembly Flash emulator — no browser plugin required. Download the latest self-hosted build from ruffle.rs and extract the files into userdata/common/arcade/bin/.

userdata/common/arcade/bin/
  ruffle.js
  ruffle_bg.wasm        ← must be served as application/wasm
  …
⚠️
MIME type required: The .wasm file must be served with the application/wasm Content-Type. Browsers will refuse to load it otherwise. Add this to your nginx config inside the arcade container:
# nginx.conf addition — add inside the server {} block:
location ~* \.wasm$ {
    types { }
    default_type application/wasm;
    add_header Cache-Control "public, max-age=86400";
}

# Or add to mime.types:
application/wasm  wasm;
Embedding Ruffle in the game page
<!-- In your game player page -->
<script>
  window.RufflePlayer = window.RufflePlayer || {};
  window.RufflePlayer.config = {
    autoplay: "on",
    unmuteOverlay: "hidden",
  };
</script>
<script src="/bin/ruffle.js"></script>
<script>
  window.addEventListener("load", () => {
    const ruffle = window.RufflePlayer.newest();
    const player = ruffle.createPlayer();
    document.getElementById("game-container").appendChild(player);
    player.load("/library/game1.swf");
  });
</script>
<div id="game-container"></div>
💾 Save File Architecture (.sol)

Flash save files are .sol (SharedObject) files. By default Ruffle writes these to the browser's local storage. To get per-user server-side saves, the arcade frontend intercepts Ruffle's save/load events and proxies them to the save endpoint instead.

# Save write:
PUT  /saves/game_name.sol
     Host: nginx serving userdata/{NAME}/arcade/saves/game_name.sol

# Save read:
GET  /saves/game_name.sol
     Host: nginx reading userdata/{NAME}/arcade/saves/game_name.sol

Ruffle exposes a callExternalInterface hook and save callbacks. Wire them up in the player initialization:

player.addEventListener("savegame", async (e) => {
  await fetch(`/saves/${e.detail.name}.sol`, {
    method: 'PUT',
    body: e.detail.data,
    headers: { 'Content-Type': 'application/octet-stream' }
  });
});

player.addEventListener("loadgame", async (e) => {
  const res = await fetch(`/saves/${e.detail.name}.sol`);
  if (res.ok) e.detail.resolve(await res.arrayBuffer());
  else e.detail.resolve(null);
});
🎮 Game Library

All .swf files live in userdata/common/arcade/library/. They are shared read-only across all users. The arcade frontend reads library.json to build the game browser UI.

userdata/common/arcade/library/
  game1.swf
  game2.swf
  game1.png             ← thumbnail (optional)
  library.json
# library.json format:
[
  {
    "id":    "game1",
    "title": "My Flash Game",
    "swf":   "library/game1.swf",
    "thumb": "library/game1.png"
  },
  ...
]

To add a game: drop the .swf (and optional thumbnail) into library/ and add an entry to library.json. No service restart required — static files are served directly by nginx.

💾 User Data Storage → SSO TAB

Save files are stored at userdata/{NAME}/arcade/saves/, where NAME is the Authentik username injected by Caddy. Each user's saves are completely isolated. This path follows the same per-user convention as other services on the server.

userdata/
  alice/
    arcade/
      saves/
        game1.sol
        game2.sol
  bob/
    arcade/
      saves/
        game1.sol       ← Bob's separate save — never touches Alice's

See the SSO tab → User Data Storage for the full cross-service user data architecture.
